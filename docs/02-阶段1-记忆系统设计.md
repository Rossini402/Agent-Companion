# 阶段 1 设计 · 记忆系统（地基）

> 来源依据：知识库 176–181（记忆系统）、179（Prompt 组装）、180（落库与抽取）
> 目标：让 Agent **记得用户** —— 跑通「对话 → 落库 → 摘要 → 长期记忆抽取 → 检索 → 组装进 Prompt」的闭环
> 技术栈（沿用课程）：Hono + Cloudflare Workers / D1 + Drizzle / Zod（contracts 单一真实来源）/ Vercel AI SDK

---

## 0. 这一阶段做什么、不做什么

| 做（v1） | 不做（留给 v2） |
|---|---|
| 三层记忆：最近原文 + 滚动摘要 + 长期记忆 | 向量检索（数据量小，先按 importance 排序） |
| 长期记忆「候选判断 → LLM 抽取」 | 模型微调 |
| 用户消息**先落库**（LLM 前） | 多会话（表结构预留，约束先锁一对一） |
| 轻量滚动摘要（字符串拼接，非 LLM） | LLM 摘要 |
| 记忆 active/disabled/deleted 可管理 | 自动遗忘策略 |

**核心原则**：服务端 D1 是上下文的唯一真实来源，前端 `messages` 只提供本轮输入；前端状态（刷新/切 Agent/多标签页）不可信。

## 1. 架构总览

```
用户发送消息
  → 校验 token
  → 找到用户拥有的 Agent → 获取/创建默认 conversation
  → ② 读取「此前」历史消息 + ③ active 长期记忆（importance desc, updatedAt desc）
       ⚠️ 实测修正：必须在「写入本轮用户消息」之前读历史，
       否则本轮消息会同时出现在 history 和 latestUserText，导致 prompt / 摘要重复。
  → ① 写入用户消息（仍在 LLM 调用前，失败也不丢）
  → ④ 组装 Prompt（人设 → 长期记忆 → 摘要 → 对象资料 → 历史 → 本轮输入）
  → ⑤ 调用上游 LLM，流式返回
  → ⑥ 流结束：写入 assistant 消息
  → ⑦ 更新 summary / message_count / last_message_at_ms
  → ⑧ 长期记忆抽取（候选判断 → LLM 抽取，异步、失败不影响回复）
```

三张表 = 三个职责层，关联但不重叠：

| 表 | 职责 | 类比 |
|---|---|---|
| `agent_conversations` | 会话容器：摘要、消息数、最近时间 | 中期记忆 |
| `agent_conversation_messages` | 完整消息流水（user/assistant） | 短期记忆 |
| `agent_memories` | 抽取出的结构化稳定信息 | 长期记忆 |

## 2. 数据表设计（D1，迁移文件 `0012_agent_chat_memory.sql`）

### 2.1 `agent_conversations` —— 会话主表

```sql
CREATE TABLE IF NOT EXISTS agent_conversations (
  id TEXT PRIMARY KEY,
  user_id TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  agent_id TEXT NOT NULL REFERENCES user_agent_companions(id) ON DELETE CASCADE,
  title TEXT,
  summary TEXT,                                  -- 滚动摘要（参与 prompt，非完整消息）
  message_count INTEGER NOT NULL DEFAULT 0,      -- 冗余统计，避免高频 count
  last_message_at_ms INTEGER,                    -- 首页列表排序
  created_at_ms INTEGER NOT NULL,
  updated_at_ms INTEGER NOT NULL
);
CREATE UNIQUE INDEX IF NOT EXISTS idx_agent_conversations_user_agent_unique
  ON agent_conversations(user_id, agent_id);     -- v1：一对一默认会话
```

**设计要点**
- `id` 独立主键：即便 v1 只有默认会话，消息表也要靠 `conversation_id` 关联 → 未来开多会话不必推翻消息表。
- `user_id + agent_id` 唯一索引 = 锁定「一个用户 × 一个 Agent = 一个会话」。
- `user_id` 必存（不能只存 `agent_id`）：每次查询都把用户作为条件，守住权限边界。

### 2.2 `agent_conversation_messages` —— 消息流水

```sql
CREATE TABLE IF NOT EXISTS agent_conversation_messages (
  id TEXT PRIMARY KEY,
  conversation_id TEXT NOT NULL REFERENCES agent_conversations(id) ON DELETE CASCADE,
  user_id TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  agent_id TEXT NOT NULL REFERENCES user_agent_companions(id) ON DELETE CASCADE,
  role TEXT NOT NULL CHECK (role IN ('user', 'assistant')),
  content TEXT NOT NULL,
  status TEXT NOT NULL CHECK (status IN ('completed', 'failed')),
  metadata_json TEXT,                            -- token/模型/provider/后续各层分析结果
  created_at_ms INTEGER NOT NULL                 -- 毫秒，分页游标
);
```

**设计要点**
- `user_id/agent_id` 冗余：查询方便 + 权限直接 + 排查清晰。
- `status` 预留 `failed`：用户消息已写但 LLM 失败时，仍保留并可重试/分析。
- `metadata_json`：先用 JSON 承接非核心字段（含阶段 2–3 的安全/意图/情绪/路由/质检结果），稳定后再升列。
- 分页游标用 `created_at_ms`；极端同毫秒冲突时升级为 `created_at_ms + id`。

### 2.3 `agent_memories` —— 长期记忆

```sql
CREATE TABLE IF NOT EXISTS agent_memories (
  id TEXT PRIMARY KEY,
  user_id TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  agent_id TEXT NOT NULL REFERENCES user_agent_companions(id) ON DELETE CASCADE,
  type TEXT NOT NULL,                            -- 偏好/边界/关系目标/对话风格/重要事实（不锁枚举，留弹性）
  content TEXT NOT NULL,                         -- 提炼短文本（≤2000），不是原文复制
  importance INTEGER NOT NULL DEFAULT 3,         -- 1–5，v1 的排序依据
  status TEXT NOT NULL CHECK (status IN ('active', 'disabled', 'deleted')),
  source_message_id TEXT REFERENCES agent_conversation_messages(id) ON DELETE SET NULL,
  created_at_ms INTEGER NOT NULL,
  updated_at_ms INTEGER NOT NULL
);
```

**设计要点**
- 记忆 ≠ 消息：消息是原文，记忆是「提炼后的稳定信息」，要能管理类型/重要度/状态。
- `importance` 是 v1 无向量检索时的注入优先级；相同则 `updated_at_ms` 更近者优先。
- `status` 三态：`active` 参与 prompt / `disabled` 数据在但不参与 / `deleted` 用户删除。
- `source_message_id`：记忆可追溯来源，用户能判断抽取是否准确。

## 4. Prompt 组装契约（核心）

注入顺序 = 一次「上下文分拣」，每段各回答一个问题：

| 段 | 回答的问题 | 放在哪 |
|---|---|---|
| Agent 人设 | 你是谁、怎么回答 | system |
| 长期记忆 | 用户有哪些稳定偏好/边界 | system |
| 会话摘要 | 更早聊过什么 | system |
| 聊天对象资料 | 这次面对的是哪个 Agent、什么关系氛围 | 单独一条 user |
| 历史消息 | 当前对话现场 | 多条 user/assistant |
| 本轮输入 | 这一次要处理什么 | 最后一条 user |

```ts
// apps/api/src/services/agent-chat/build-prompt.ts （骨架）
export function buildAgentChatMessages(input: {
  agentDefaultPrompt?: string
  activeMemories: { type: string; importance: number; content: string }[]
  summary?: string | null
  conversation: InboxChatConversation
  history: { role: 'user' | 'assistant'; content: string }[]
  latestUserText: string
}): ChatCompletionMessage[] {
  const { agentDefaultPrompt, activeMemories, summary, conversation, history, latestUserText } = input

  const messages: ChatCompletionMessage[] = [{
    role: 'system',
    content: [
      agentDefaultPrompt || '你是 AI Agent Web 控制台里的聊天陪伴助手。',
      '请基于当前聊天对象、关系氛围和用户意图，用简洁、自然的中文回答。',
      '你的建议应尊重双方边界，避免操控式话术、制造焦虑或诱导过度解读。',
      activeMemories.length
        ? ['以下是用户与该 Agent 的长期记忆，请优先尊重：',
           ...activeMemories.map(m => `- [${m.type} / 重要度 ${m.importance}] ${m.content}`)].join('\n')
        : '',
      summary ? `此前对话摘要：${summary}` : '',
    ].filter(Boolean).join('\n'),
  }]

  // 聊天对象静态资料：单独 user message
  messages.push({ role: 'user', content: [
    `聊天对象：${conversation.name}（${conversation.handle}）`,
    `关系阶段：${conversation.relationship}`,
    `聊天主题：${conversation.headline}`,
    `对象备注：${conversation.profileNote}`,
  ].join('\n') })

  // 历史（来自 D1，清洗空消息，保留原 role）
  for (const m of history) {
    const text = normalizeStoredMessage(m.content)
    if (text) messages.push({ role: m.role, content: text })
  }
  if (history.length && latestUserText) {
    messages.push({ role: 'user', content: latestUserText })
  }
  return messages
}
```

## 5. 落库 / 摘要 / 抽取（回复后处理）

### 5.1 用户消息先落库（LLM 前）
```ts
if (agentId && conversationId && latestUserText) {
  sourceUserMessageId = crypto.randomUUID()
  const nowMs = Date.now()
  await insertAgentConversationMessage({ id: sourceUserMessageId, conversationId, userId, agentId,
    role: 'user', content: latestUserText, status: 'completed', nowMs })
  await updateAgentConversationAfterMessage({ userId, agentId, conversationId,
    summary: prev.summary ?? null, messageCount: (prev.messageCount ?? 0) + 1,
    lastMessageAtMs: nowMs, nowMs })
}
```
> 失败也不丢用户消息；`sourceUserMessageId` 供后续长期记忆 `source_message_id` 关联。

### 5.2 轻量滚动摘要（非 LLM）
> 既有 summary **已累积此前所有对话脉络**，每轮只需把「本轮新增的一来一回」追加到尾部 → 压缩空白 → 取**末尾约 1600 字符**（保留最近内容）。
> ⚠️ 实测修正：**不要**再把 recent 历史窗口重新拼进来——它和 previousSummary 重叠，会造成同句重复计数。
> 选择字符串策略而非 LLM 摘要：零额外推理成本、确定性、够用即可。

### 5.3 长期记忆抽取（候选判断 → LLM 抽取，异步）
```
本轮内容
  → fast reject（本地规则）：短寒暄 / 一次性情绪 / 重复 / 敏感信息 → 直接丢弃
  → 候选判断（LangChain 结构化输出 + 关键词兜底）：是否值得记
  → 命中才调 LLM 抽取 → 归类 type + 给 importance + content 提炼
  → 写入 agent_memories(status='active', source_message_id=本轮用户消息)
```
> 异步执行，失败不影响主回复；不是所有对话都值得存——**先拒后抽**避免记忆库被污染。

## 6. API 端点（v1）

| 方法 | 路径 | 说明 |
|---|---|---|
| POST | `/agent/chat` | 发送消息：落库→组装→流式返回→后处理 |
| GET | `/agent/conversations/:agentId` | 恢复历史（含 summary/openingMessage/nextCursor） |
| GET | `/agent/conversations/:agentId/messages?cursor=` | 分页加载更早消息 |
| GET | `/agent/memories?agentId=` | 记忆库列表（active/disabled） |
| PATCH | `/agent/memories/:id` | 改 type/content/importance/status（受 `UpdateAgentMemoryRequestSchema` 约束） |
| DELETE | `/agent/memories/:id` | 软删除 → status='deleted' |

## 8. 验证（按规范：改字段/状态流转必须真实联调）

1. **落库不丢**：发消息后立刻断开上游 LLM（mock 抛错）→ 刷新页面，用户消息仍在、status 体现失败态。
2. **上下文以服务端为准**：发消息 → 改前端 `messages` 为空 → 再发 → 回复仍延续历史（证明用 D1 而非前端状态）。
3. **记忆注入**：手动插一条 `偏好/importance=5` 记忆 → 新对话回复风格遵守该偏好。
4. **排序正确**：插多条不同 importance 记忆 → 检查注入顺序 importance desc, updatedAt desc。
5. **候选先拒**：发「嗯」「哈哈」→ 不应产生长期记忆；发「以后回复我直接一点」→ 应抽取为偏好。
6. **摘要滚动**：连续 >8 轮 → summary 末尾保留最近内容、长度受 1600 字符约束。
