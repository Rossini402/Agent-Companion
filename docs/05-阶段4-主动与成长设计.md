# 阶段4 · 主动与成长

> 来源依据：知识库 189（用户反馈闭环）、190（主动关怀系统）；严丝合缝衔接阶段1 落地代码（`agent-chat.ts` / `build-prompt.ts` / `repository.ts` / `schema/agent-chat.ts`）。
> 一句话目标：让 Agent **越来越懂你 + 会主动出现**——用户点赞/点踩入表并在下一轮注入 prompt（反馈闭环），用户手动生成的关怀消息以真实 assistant 消息写入聊天历史（主动陪伴），两者都不破坏沉浸感。

## 做 / 不做（v1 边界）

| 做（v1） | 不做（留给 v2） |
|---|---|
| 反馈表 `agent_message_feedbacks`：一用户 × 一消息唯一一条，可点赞↔点踩切换（upsert）（189） | 复杂评分体系 / 模型微调（189） |
| 历史消息左连接回显已提交的反馈状态（189） | 自动把多条反馈归纳成"用户偏好画像"（189 后续升级） |
| 最近 N 条反馈格式化注入 system prompt（正向=偏好风格，负向=需避免），明令"不要在回复里提评分/点赞/反馈"（189） | 负反馈触发质检复盘 / 自动改 Agent 人设 prompt（189 后续） |
| 关怀两表 `agent_care_plans` / `agent_care_events`，一用户 × 一 Agent 唯一一份计划（190） | Cloudflare Cron 自动定时生成（先存 `next_run_at_ms` 占位）（190） |
| 手动生成关怀消息：模板文案 → 写入 `agent_conversation_messages`（role=assistant，metadata 标 `source:proactive_care`）→ 写 `agent_care_events`（190） | LLM 动态生成关怀文案（保持外层流程不变，先模板）（190） |
| 关怀事件未读/已读状态：打开聊天即标已读（190） | 浏览器/邮件/短信/微信等外部推送（190） |
| 反馈 reason 枚举收口（good_tone/too_long/…），但 v1 UI 只传 rating（189） | 撤销反馈 / 清除偏好 / 反馈二级弹窗（189 后续） |

**核心原则继承阶段1**：服务端 D1 是唯一真实来源；增强能力（反馈、关怀未读）必须对主链路**容错**——新表不可用时聊天与首页不能崩（190）。

## 架构与 pipeline

阶段4 有两条独立链路，分别挂在 stage1 不同位置。

### 链路A · 反馈闭环（读侧注入 + 旁路写入）

挂在 `/chat` 主链路的 ②③ 读取段与 ④ 组装段之间；写入是独立端点，不在 `/chat` 内。

```
POST /agent/chat（stage1 主链路，仅在 ②③④ 之间增一步）
  ① 用户消息先落库（不变）
  ②③ 读历史 + active 长期记忆（不变）
  ②.5 【新增】读最近 N 条反馈 listRecentFeedbacks(userId, agentId, limit)
  ④ 组装 prompt ──【改动点】buildAgentChatMessages 多收 recentFeedbacks，
       在 system 段「长期记忆 / 摘要」之后追加「反馈段」
  ⑤⑥⑦⑧ 流式生成 / assistant 落库 / 摘要 / 记忆抽取（全不变）

POST /agent/messages/:messageId/feedback（独立端点，与 /chat 解耦）
  校验：登录用户 + 该 assistant 消息属于本人本 Agent + status=completed
  → upsertFeedback（按 user_id+message_id 唯一索引：有则 update，无则 insert）

GET /agent/conversations/:agentId（恢复历史，stage1 已有）
  → 查询消息时 leftJoin agent_message_feedbacks，回显每条 assistant 的 feedback
```

### 链路B · 主动关怀（旁路生成，复用 stage1 落库函数）

完全不经过 `/chat`，是一条独立"生成关怀消息"链路，但**复用 stage1 的会话/落库函数**，让关怀消息混入同一条聊天历史。

```
POST /agent/care/:agentId/generate（手动触发，无 LLM）
  校验用户 + Agent 归属
  → findOrCreateCarePlan（无则建默认：disabled/daily/21:30/[long_absence,night]/gentle）
  → 选 scene（入参 > plan.scenes[0] > 兜底 long_absence）
  → getOrCreateConversation（复用 stage1 repository）
  → buildProactiveCareMessage（模板，代码规则）
  → insertMessage(role='assistant', metadata_json={source:'proactive_care',...})（复用 stage1）
  → insertCareEvent(status='generated', readAtMs=null)
  → updateConversationAfterMessage（messageCount+1，复用 stage1；summary 原样透传不动）
  → 返回 event

GET /agent/conversations/:agentId（stage1 已有，进入聊天前加一步）
  →【改动点】markCareEventsRead(userId, agentId)  // try/catch 容错
  → listAgentConversationMessages（关怀消息天然出现在历史里，无需特判）
```

> 关键抽象（190）：主动关怀**不是通知系统，而是聊天系统的一种消息来源**。所以必须写 `agent_conversation_messages`，否则它只是后台事件，无法成为后续记忆/摘要/反馈的上下文。

## 节点设计（逐节点：输入/输出 · LLM还是代码 · 失败兜底）

### A1 提交反馈 `submitFeedback`
- 输入：`userId`、`agentId`、`messageId`、`{rating, reason?, note?}`。
- 处理：**代码规则**（无 LLM）。先 `findMessageForFeedback` 校验该消息是当前用户当前 Agent 的 assistant 消息且 `status='completed'`；再 `upsertFeedback`（按 `user_id+message_id` 唯一索引切换/更新）。
- 输出：`{rating, reason, note, updatedAtMs}`。
- 失败兜底：消息不存在/越权 → 404；唯一索引冲突由 upsert 吸收（先查再 update/insert，避免依赖 D1 原生 upsert，便于保留 `created_at_ms`）。

### A2 读取最近反馈并注入 `listRecentFeedbacks` + `getFeedbackSystemInstruction`
- 输入：`userId`、`agentId`、`limit`（建议 6）。
- 处理：**代码规则**。按 `updated_at_ms desc` 取最近 N 条 → 格式化为系统指令；空则返回空串（不污染 prompt）。
- 输出：拼进 system content 的一段文本。
- 失败兜底：查询异常 try/catch 吞掉返回 `[]`，反馈段缺失不影响主回复（与 stage1「记忆抽取失败不影响回复」一致）。

### B1 查找/创建关怀计划 `findOrCreateCarePlan`
- 输入：`userId`、`agentId`。
- 处理：**代码规则**。无则建默认计划（`enabled=false`）；读 `scenes_json` 后做**白名单过滤**（JSON 字段无 enum 约束，读出来要再清洗）（190）。
- 输出：`AgentCarePlan`。
- 失败兜底：解析 `scenes_json` 失败 → 退回默认场景 `['long_absence','night']`。

### B2 选场景 + 生成文案 `buildProactiveCareMessage`
- 输入：`agentName`、`scene`、`tone`、`customPrompt`。
- 处理：**代码规则模板**（v1 不调 LLM——主动关怀要稳定可用，不应依赖用户是否配了第三方 LLM）（190）。`customPrompt` 优先，否则按 scene 取模板，统一加 tone 前缀，`slice(0,1000)`。
- 输出：关怀文案字符串。
- 失败兜底：scene 不在表内 → 兜底 `long_absence` 文案。
- v2 退路：仅替换本函数内部为 LLM 润色，外层落库/事件流程不变（190）。

### B3 写消息 + 写事件 + 标已读
- `insertMessage`：复用 stage1，`metadata_json.source='proactive_care'`（便于 UI 加标识、统计区分主动/被动）（190）。
- `insertCareEvent`：`status='generated'`，绑定 `conversation_id`+`message_id`。
- `markCareEventsRead`：打开聊天时把该 Agent `generated` 且 `readAtMs is null` 的事件批量置 `read`；**整段 try/catch**，新表未迁移时聊天历史接口照常返回（190）。

## Prompt 注入（与 build-prompt.ts 衔接）

只改 `buildAgentChatMessages`：多收一个 `recentFeedbacks`，在 system 段「长期记忆 → 摘要」之后追加「反馈段」。位置在 system 内、对象资料/历史之前——保持 stage1「人设→记忆→摘要→对象→历史→本轮」的分拣顺序，反馈属于"用户偏好校准"，与记忆同层。

```ts
// build-prompt.ts 改动点①：类型多收反馈
export type RecentFeedback = { rating: "positive" | "negative"; reason: string | null; content: string }

export function buildAgentChatMessages(input: {
  agentDefaultPrompt?: string | null
  activeMemories: ActiveMemory[]
  summary?: string | null
  recentFeedbacks?: RecentFeedback[]   // ← 新增
  conversation: InboxChatConversation
  history: HistoryMessage[]
  latestUserText: string
}): ChatCompletionMessage[] {
  const { recentFeedbacks = [], /* …其余解构不变… */ } = input

  const systemContent = [
    agentDefaultPrompt || "你是 AI Agent Web 控制台里的聊天陪伴助手。",
    "请基于当前聊天对象、关系氛围和用户意图，用简洁、自然的中文回答用户。",
    "如果用户要求起草回复，请直接给出可发送的聊天内容，避免正式公文格式和职场汇报语气。",
    "你的建议应尊重双方边界，避免操控式话术、制造焦虑或诱导过度解读。",
    activeMemories.length > 0
      ? ["以下是用户与该 Agent 的长期记忆，请优先尊重：",
         ...activeMemories.map((m) => `- [${m.type} / 重要度 ${m.importance}] ${m.content}`)].join("\n")
      : "",
    summary ? `此前对话摘要：${summary}` : "",
    // ── 改动点②：反馈段（189）──
    recentFeedbacks.length > 0
      ? [
          "近期用户对该 Agent 回复的反馈：",
          ...recentFeedbacks.map(
            (f) => `- [${f.rating === "positive" ? "正向" : "负向"}${f.reason ? "/" + f.reason : ""}] ${f.content}`,
          ),
          "请把正向反馈视为用户偏好的表达风格，把负向反馈视为需要避免的问题；不要在回复中提到评分、点赞、点踩或反馈记录。",
        ].join("\n")
      : "",
  ].filter(Boolean).join("\n")
  // …下方对象资料 / 历史 / 本轮输入完全不变…
}
```

## 代码骨架（可直接粘贴）

### 反馈 repository（追加 `apps/api/src/services/agent-chat/feedback.ts`）

```ts
import { and, desc, eq } from "drizzle-orm"
import { type Db, agentMessageFeedbacks, agentConversationMessages } from "@ai-companion/db"
import type { RecentFeedback } from "./build-prompt"

/** 校验：该消息是当前用户当前 Agent 的 assistant 消息且 completed，返回其 conversationId */
export async function findMessageForFeedback(
  db: Db,
  args: { userId: string; agentId: string; messageId: string },
): Promise<{ id: string; conversationId: string } | null> {
  const rows = await db
    .select({ id: agentConversationMessages.id, conversationId: agentConversationMessages.conversationId })
    .from(agentConversationMessages)
    .where(
      and(
        eq(agentConversationMessages.id, args.messageId),
        eq(agentConversationMessages.userId, args.userId),
        eq(agentConversationMessages.agentId, args.agentId),
        eq(agentConversationMessages.role, "assistant"),
        eq(agentConversationMessages.status, "completed"),
      ),
    )
    .limit(1)
  return rows[0] ?? null
}

/** upsert：先查后 update/insert（保留 created_at_ms），契合唯一索引 user_id+message_id */
export async function upsertFeedback(
  db: Db,
  args: {
    userId: string; agentId: string; conversationId: string; messageId: string
    rating: "positive" | "negative"; reason: string | null; note: string | null; nowMs: number
  },
): Promise<void> {
  const existing = await db
    .select({ id: agentMessageFeedbacks.id })
    .from(agentMessageFeedbacks)
    .where(and(eq(agentMessageFeedbacks.userId, args.userId), eq(agentMessageFeedbacks.messageId, args.messageId)))
    .limit(1)
  if (existing[0]) {
    await db.update(agentMessageFeedbacks)
      .set({ rating: args.rating, reason: args.reason, note: args.note, updatedAtMs: args.nowMs })
      .where(eq(agentMessageFeedbacks.id, existing[0].id))
    return
  }
  await db.insert(agentMessageFeedbacks).values({
    id: crypto.randomUUID(), userId: args.userId, agentId: args.agentId,
    conversationId: args.conversationId, messageId: args.messageId,
    rating: args.rating, reason: args.reason, note: args.note,
    createdAtMs: args.nowMs, updatedAtMs: args.nowMs,
  })
}

/** 注入用：最近 N 条反馈（容错——异常返回空，不拖垮主回复） */
export async function listRecentFeedbacks(
```

```ts
export function buildProactiveCareMessage(p: { scene: CareScene; tone: CareTone; customPrompt: string | null }): string {
  const prefix = TONE_PREFIX[p.tone]

  if (custom) return `${prefix}。${custom}`.slice(0, 1000)

/** 计算下一次触发时间（v1 仅存档，留给 Cron） */
export function calculateNextCareRunAtMs(p: { enabled: boolean; frequency: "daily" | "weekly" | "custom"; preferredTime: string | null; nowMs: number }): number | null {
  if (!p.enabled) return null
  const next = new Date(p.nowMs)
  if (p.preferredTime) {
    const [h, m] = p.preferredTime.split(":").map(Number)
    if (Number.isFinite(h) && Number.isFinite(m))
      next.setHours(Math.min(23, Math.max(0, h)), Math.min(59, Math.max(0, m)), 0, 0)
  }
  if (next.getTime() <= p.nowMs) next.setDate(next.getDate() + (p.frequency === "weekly" ? 7 : 1))
  return next.getTime()
}
```

## 结构化输出依赖与退路

**当前 v1：两条链路都不依赖 LLM 结构化输出**——反馈是用户显式点击（代码规则），关怀文案是模板（代码规则）。这是有意为之（189/190 均强调 MVP 不接 LLM，先证数据闭环）。因此 `response_format:json_object` 在阶段4 无硬依赖，零结构化风险。

**v2 才会引入 LLM 的两处，及其退路**（提前规划，沿用 stage1「记忆抽取失败不影响主回复」的容错哲学）：

1. **关怀文案 LLM 润色**（替换 `buildProactiveCareMessage` 内部）：输出是**纯文本**，不需要 JSON——天然规避 json_object 不稳问题。退路最简单：LLM 调用失败/超时 → 回落到本函数现有模板。语气约束写进 prompt（不要显得在监控用户、不要情感绑架、不要频繁索取回应、不说"我一直在等你"）（190）。
2. **负反馈归纳为偏好画像 / 反馈 reason 自动分类**（189 后续）：这才需要结构化输出（`{type, importance, content}`，与 stage1 记忆抽取同形）。**若 deepseek-v4-flash 的 `response_format:json_object` 不稳，退路三连**：
   - **Prompt 约束**：system 中明令「只输出 JSON 数组，每项 `{type,importance,content}`，不要 markdown 代码块、不要解释」。
   - **zod 解析 + 容错抽取**：先 `JSON.parse`；失败则用正则抠出首个 `[...]`/`{...}` 片段再 parse；再用 `z.array(ExtractedMemorySchema).safeParse` 校验，剔除非法项。
   - **有限重试**：`safeParse` 失败时最多重试 1 次（追加「上次输出非合法 JSON，请仅返回 JSON」），仍失败则**返回空数组**（与 stage1 记忆抽取 `catch` 后不写入一致，绝不阻塞主链路）。
   - 注意 deepseek-v4-flash 是推理模型，流式只取 `delta.content`；做结构化抽取应走**非流式**一次性返回再 parse（现有 `streamDeepSeekChat` 是 generator，需另加一个非流式 helper，把所有 delta 累积后整体 parse）。
