# 阶段2 · 对话理解链

> 来源依据：知识库 183 意图识别、184 情绪路由、185 回复策略模板、187 关系阶段系统、188 记忆候选判断
> 一句话目标：在 LLM 生成回复**之前**插入一条「意图 → 情绪 → 关系阶段 → 情绪路由 → 回复策略」的理解链，**LLM 负责理解语义、代码负责选择策略**，结果注入 system prompt 并写入用户消息的 `metadata_json`，让 Agent 从「能回答」变成「会回应」。
> 严丝合缝基准：沿用阶段1自写 DeepSeek 流式（**不引入 LangChain/LangGraph**），结构化输出走 DeepSeek OpenAI 兼容的 `response_format:{type:'json_object'}` + Zod 解析 + 重试。

---

## 1. 这一阶段做什么、不做什么

| 做（v1） | 不做（留给后续） |
|---|---|
| 5 个理解节点：意图（LLM）→ 情绪（LLM）→ 关系阶段（LLM）→ 情绪路由（代码）→ 回复策略（代码） | 引入 LangChain `withStructuredOutput` / LangGraph 编排（本项目栈是自写 fetch，保持一致） |
| LLM 节点用 `response_format:json_object` + Zod 解析 + 多次重试 | 安全边界判断（文章中 safety 在 intent 之前；本项目阶段1无安全层，v1 留兜底接口，不实现拦截） |
| 每个 LLM 节点都有 `fallback*` 兜底，失败不影响主回复 | 多轮情绪趋势、路由 A/B、效果评估 |
| 代码归一化（`normalize*`）治理 LLM 输出，守住产品策略 | 关系阶段独立持久化表（`agent_relationship_states`），v1 只写 metadata |
| 理解结果写入**用户消息** `metadata_json`（`analysisVersion: conversation-understanding-v2`） | 前端/后台可视化 metadata（属于展示层，本阶段只改 API） |
| 理解结果注入 system prompt（人设之后、长期记忆之前） | 记忆候选判断节点接入理解链（188 属于回复**后**处理，本阶段在「数据模型」「实施清单」预留，主链路实现见阶段3或本阶段尾段可选项） |
| 串行执行（intent→emotion→stage 各一次 LLM 调用，复用同一 DeepSeek 配置） | 并行 / 单次 prompt 合并多任务（先稳，后优化成本） |

**核心原则**（与阶段1一致）：
- 理解链在**服务端**做，前端不可信（用户可绕过前端直接打 API）。
- LLM 判断语义，代码归一化兜底；涉及置信度、依赖风险、边界时**不完全信任模型**。
- 任何理解节点失败都回退到保守兜底，**绝不让一轮聊天失败**。
- 不在回复中暴露内部标签（intent/route/policy 名称），它们只是隐性策略。

## 2. 架构与 pipeline：插入点

阶段1 `/chat` 主链路（`agent-chat.ts`）当前是：

```
①用户消息先落库 → ②③读历史/记忆 → ④buildAgentChatMessages → ⑤DeepSeek 流式
  → ⑥assistant 落库 → ⑦滚动摘要 → ⑧记忆抽取
```

阶段2 把「对话理解链」插在 **②③ 读历史/记忆之后、④ 组装 prompt 之前**，并调整用户消息落库时机：

```
用户发消息
  → getOrCreateConversation
  → ②③ 读「此前」history + activeMemories（保持阶段1顺序，先读再写）
  → 【新增 ★ 对话理解链 analyzeConversationUnderstanding(...)】   ← 本阶段插入点
        ┌─ classifyIntent       (LLM·json_object)  意图
        ├─ detectEmotion        (LLM·json_object)  情绪（输入含 intent）
        ├─ analyzeRelationshipStage (LLM·json_object) 关系阶段（输入含 intent+emotion+messageCount+summary）
        ├─ routeEmotion         (代码规则)         情绪路由（输入 intent+emotion+stage）
        └─ buildReplyPolicy     (代码规则)         回复策略（输入 intent+emotion+stage+route）
        失败 → 整链兜底（fallbackIntent/Emotion/Stage + 规则推导 route/policy）
  → ① 用户消息落库（content + metadataJson=理解结果）           ← 落库时机后移到理解之后
  → 读 Agent 默认人设
  → ④ buildAgentChatMessages（system 段新增 intent/route/policy/stage 注入）  ← 改动点
  → ⑤ DeepSeek 流式生成
  → ⑥ assistant 落库 → ⑦ 摘要 → ⑧ 记忆抽取（阶段1原样）
```

**与阶段1的关键衔接 / 改动**：

1. **用户消息落库时机后移**（重要取舍）。阶段1是「用户消息先落库（LLM 调用前，失败也不丢）」。理解结果要写进**用户消息**的 `metadata_json`（文章 183/184/185/187 都是写在 user message 上）。为同时满足「先落库不丢」与「带分析结果」，v1 选择：**理解链在用户消息落库前执行，落库时一次性带上 metadata**。理解链本身全程有兜底（最坏拿到 fallback），不会阻塞落库，因此不破坏「失败也不丢」原则——只是把落库点从「LLM 前」移到「理解链后、生成前」。详见 §节点设计末「落库时机」与 problems。
2. 理解链使用与最终聊天**同一个 DeepSeek 配置**（model/baseUrl/apiKey），保证模型行为与用户配置一致（文章 183「意图判断使用和最终聊天同一个用户配置的 LLM」）。
3. 理解链顺序严格遵循文章 187：`intent → emotion → relationshipStage → route → replyPolicy`。

## 3. 节点设计（逐节点：输入/输出、LLM 还是代码、失败兜底）

公共输入上下文 `UnderstandingContext`：

```ts
{
  config: DeepSeekConfig            // 复用阶段1的 { apiKey, model, baseUrl }
  agentName: string                 // = conversation.name
  agentGuardrails: string | null    // = agentRow.default_prompt（v1 复用人设，无独立 guardrails 字段）
  activeMemories: ActiveMemory[]    // 阶段1 listActiveMemories 的结果
  recentMessages: HistoryMessage[]  // 阶段1 listRecentMessages 的结果
  conversationSummary: string | null
  messageCount: number
  userText: string
  signal?: AbortSignal              // = c.req.raw.signal
}
```

### 3.1 classifyIntent —— 意图识别（LLM）
- **输入**：agentName、agentGuardrails、activeMemories、recentMessages、userText。
- **输出**：`ConversationIntent`（primary/secondary/confidence/userNeed/requestedAgentAction/relationshipSignal/replyExpectation/shouldClarify/clarifyingQuestion/promptGuidance）。（文章183）
- **LLM 还是代码**：LLM（语义理解）+ 代码 `normalizeConversationIntent`（次要意图去重、低置信度<0.45 降级 unclear、memory_update/preference_setting/agent_feedback 收口）。
- **失败兜底**：`fallbackIntent`（primary=unclear, confidence=0.3, requestedAgentAction=ask_follow_up, shouldClarify=true）。userText 为空时返回「轻柔延续」意图（casual_chat/feel_connected）。

### 3.2 detectEmotion —— 情绪识别（LLM）
- **输入**：上 + `intent`（同情绪在不同意图下走不同线，文章184）。
- **输出**：`ConversationEmotion`（primaryEmotion/secondaryEmotions/intensity/valence/arousal/needsComfort/needsDeescalation/needsClarification/emotionalCue/replyTone）。
- **LLM 还是代码**：LLM + 代码 `normalizeConversationEmotion`（去重次要情绪；强负面 intensity≥0.7→needsComfort；angry/hurt+high arousal→needsDeescalation）。
- **失败兜底**：`fallbackEmotion`（neutral/中性陪伴/needsClarification=true）。

### 3.3 analyzeRelationshipStage —— 关系阶段（LLM）
- **输入**：上 + `intent` + `emotion` + `conversationSummary` + `messageCount`（文章187 强调结合消息数与历史）。
- **输出**：`ConversationRelationshipStage`（stage/displayName/closenessScore/trustLevel/stability/boundaryMode/intimacyPermission/pacing/riskSignals/relationshipGuidance）。
- **LLM 还是代码**：LLM + 代码 `normalizeRelationshipStage`（历史<6 条强拉回 new_connection、closenessScore≤35、intimacyPermission=low；relationshipSignal=dependency_risk→dependency_watch；conflict/feeling_hurt/hurt/disappointed→repairing）。
- **失败兜底**：启发式 `buildHeuristicRelationshipStage`（按 messageCount+memory importance+亲近信号算 closenessScore，文章187 的阈值表：≥80&≥75→close_bond；≥36&≥58→trusted_companion；≥16&≥38→comfortable_chat；≥6→warming_up；else→new_connection）。**这是唯一一个「兜底也带逻辑」的节点**，因为关系阶段可纯靠 messageCount 推。

### 3.4 routeEmotion —— 情绪路由（**代码规则**）
- **输入**：intent + emotion + relationshipStage（路由是产品策略，文章184「LLM 负责识别情绪，代码负责选择路由」）。
- **输出**：`EmotionRoute`（route/responseLength/shouldAskQuestion/shouldGiveAdvice/shouldUsePetName/shouldMirrorEmotion/routeGuidance）。
- **规则要点**（优先级从高到低，文章184+187）：
  1. `relationshipStage.stage===repairing || pacing===repair_first` → `relationship_repair`。
  2. `stage∈{boundary_sensitive,dependency_watch} || boundaryMode===firm` → `calm_deescalation`。
  3. `emotion.needsDeescalation || primaryEmotion===angry` → repair 类则 `relationship_repair` 否则 `calm_deescalation`。
  4. `intent.primary∈{conversation_repair,agent_feedback}` → `relationship_repair`。
  5. `intent.primary===romantic_flirt || emotion===affectionate` → `playful_flirt`；**但** stage===new_connection 或 intimacyPermission===low 时降级 `light_companion`（187）。
  6. `intent.primary===relationship_advice || requestedAgentAction===analyze_situation` → needsComfort 则 `warm_comfort` 否则 `practical_support`。
  7. `emotion.needsComfort || valence===negative` → tired/companionship_presence 则 `quiet_presence` 否则 `warm_comfort`。
  8. default → `light_companion`。
- **失败兜底**：无 LLM 调用，不会失败；intent+emotion 都缺时返回 `fallbackEmotionRoute`（gentle_clarification）。

### 3.5 buildReplyPolicy —— 回复策略（**代码规则**）
- **输入**：safety(占位) + intent + emotion + relationshipStage + route。
- **输出**：`ReplyPolicy`（policy/sentenceBudget/rhythm/openingMove/allowedMoves/forbiddenMoves/questionLimit/adviceLimit/intimacyLevel/styleGuidance）。
- **规则要点**（文章185+187）：先 `sentenceBudgetForRoute(route)` 由 responseLength 推句数预算；按 `route.route` 进 switch 选 policy/动作集；memory_update/preference_setting → `memory_ack`（≤2句、只 acknowledge_memory）；二次修正：intimacyPermission===low→禁 intense_flirt；pacing===slow_down→压句数/禁过早建议；pacing===repair_first→policy=relationship_repair；强负面 intensity≥0.75→禁 intense_flirt+premature_advice；`!route.shouldAskQuestion`→questionLimit=0；`!route.shouldGiveAdvice`→adviceLimit=0。
- **失败兜底**：上游全缺 → `fallbackReplyPolicy`（gentle_clarify/1问0建议）。

> 落库时机：以上 5 节点产出 `{intent, emotion, relationshipStage, route, replyPolicy}` 后，**才**写用户消息（带 metadata）。理解链整体 try/catch，catch 分支用 fallback 组装一套完整结果（见 §7 `analyzeConversationUnderstanding`），保证 metadata 永远完整、落库永不阻塞。

## 6. Prompt 注入（与 build-prompt.ts 衔接）

阶段1 `buildAgentChatMessages` 的 system 段是一个 `[...].filter(Boolean).join("\n")` 数组。注入位置遵循文章 183/184/185/187：**人设之后、长期记忆之前**，顺序 safety→intent→relationshipStage→emotion+route→replyPolicy。

**改动点 1**：`build-prompt.ts` 入参新增 `understanding?: ConversationUnderstanding | null`。

**改动点 2**：system 数组里插入 4 段（各自空值返回空串，被 `filter(Boolean)` 吃掉）：

```ts
// build-prompt.ts —— system 段（节选，★ 为新增）
```

### 7.1 结构化输出基建 `understanding/structured.ts`（替代文章的 LangChain，适配本项目自写 fetch）

```ts
import type { ZodSchema } from "zod"
import type { DeepSeekConfig } from "../../llm/deepseek"

type Msg = { role: "system" | "user"; content: string }

/**
 * DeepSeek json_object 结构化输出 + Zod 解析 + 重试。
 * 退路链：response_format:json_object → 解析失败则提取首个 {...} → 仍失败抛错由上层兜底。
 * 注意：deepseek-v4-flash 为推理模型，需从 message.content 取最终文本（非 delta，这里非流式）。
 */
export async function callStructured<T>(args: {
  config: DeepSeekConfig
  schema: ZodSchema<T>
  system: string
  user: string
  signal?: AbortSignal
  maxRetries?: number
}): Promise<T> {
  const { config, schema, system, user, signal } = args
  const maxRetries = args.maxRetries ?? 2
  const baseUrl = (config.baseUrl ?? "https://api.deepseek.com").replace(/\/$/, "")
  let lastErr: unknown
  for (let attempt = 0; attempt <= maxRetries; attempt++) {
    try {
      const messages: Msg[] = [
        { role: "system", content: system + "\n你必须只输出一个合法 JSON 对象，不要任何解释、不要 markdown 代码块。" },
        { role: "user", content: user },
      ]
      const res = await fetch(`${baseUrl}/chat/completions`, {
        method: "POST",
        headers: { "Content-Type": "application/json", Authorization: `Bearer ${config.apiKey}` },
        body: JSON.stringify({
          model: config.model,
          messages,
          stream: false,
          temperature: 0,                       // 判断类任务用 0，稳定
          response_format: { type: "json_object" }, // 退路①：原生 JSON mode
        }),
        signal,
      })
      if (!res.ok) throw new Error(`structured_http_${res.status}: ${(await res.text()).slice(0, 300)}`)
      const json = (await res.json()) as { choices?: { message?: { content?: string } }[] }
      const raw = json.choices?.[0]?.message?.content ?? ""
      return schema.parse(extractJson(raw))   // 退路②：容错提取 + Zod 解析（解析失败→重试）
    } catch (err) {
      lastErr = err
    }
  }
  throw lastErr instanceof Error ? lastErr : new Error("structured_output_failed")
}

/** 退路②：若模型多吐了 markdown / 前后缀，截取首个完整 JSON 对象 */
function extractJson(raw: string): unknown {
  const trimmed = raw.trim().replace(/^```(?:json)?/i, "").replace(/```$/, "").trim()
  try { return JSON.parse(trimmed) } catch { /* fallthrough */ }
  const start = trimmed.indexOf("{")
  const end = trimmed.lastIndexOf("}")
  if (start >= 0 && end > start) return JSON.parse(trimmed.slice(start, end + 1))
  throw new Error("no_json_object_found")
}
```

```ts
export async function classifyIntent(ctx: UnderstandingContext): Promise<ConversationIntent> {
  const text = ctx.userText.trim()
  if (!text) return normalizeIntent({ ...fallbackIntent, primary: "casual_chat", confidence: 0.7,
    userNeed: "feel_connected", requestedAgentAction: "continue_topic", shouldClarify: false,
    clarifyingQuestion: null, promptGuidance: "用户没有提供明确新内容时，轻柔延续当前话题，不要制造压力。" })
  try {
    const user = [
      `Agent 名称：${ctx.agentName || "未命名 Agent"}`,
      `Agent 边界规则：\n${ctx.agentGuardrails || "暂无"}`,
      `长期记忆：\n${formatMemories(ctx.activeMemories)}`,
      `最近对话：\n${formatRecent(ctx.recentMessages)}`,
      `本轮用户输入：\n${text}`,
    ].join("\n\n")
    const result = await callStructured({ config: ctx.config, schema: ConversationIntentSchema, system: SYSTEM, user, signal: ctx.signal })

/** 代码归一化（文章183）：守住产品策略 */
export function normalizeIntent(intent: ConversationIntent): ConversationIntent {
  const next: ConversationIntent = {
    ...intent,
    secondary: Array.from(new Set(intent.secondary.filter((s) => s !== intent.primary))).slice(0, 3),
    replyExpectation: { ...intent.replyExpectation },
    clarifyingQuestion: intent.clarifyingQuestion?.trim() || null,
    promptGuidance: intent.promptGuidance.trim() || fallbackIntent.promptGuidance,
  }
  if (next.confidence < 0.45) {
    next.primary = "unclear"; next.secondary = []; next.userNeed = "unknown"
    next.requestedAgentAction = "ask_follow_up"; next.shouldClarify = true
    next.replyExpectation.shouldAskQuestion = true
  }
  if (next.primary === "memory_update") {
    next.userNeed = "update_memory"; next.requestedAgentAction = "remember_fact"
    next.replyExpectation.depth = "short"; next.replyExpectation.shouldAskQuestion = false; next.shouldClarify = false
  }
  if (next.primary === "preference_setting" || next.primary === "agent_feedback") {
    next.userNeed = "adjust_agent"
    next.requestedAgentAction = next.primary === "agent_feedback" ? "repair_misunderstanding" : "adjust_style"
  }
  if (next.shouldClarify && !next.clarifyingQuestion) next.clarifyingQuestion = fallbackIntent.clarifyingQuestion
  return next
}
```

### 7.3 编排入口 `understanding/index.ts`（串行，替代 LangGraph）

```ts
import type { ConversationUnderstanding } from "@ai-companion/contracts"
import type { UnderstandingContext } from "./types"
import { classifyIntent, fallbackIntent, normalizeIntent } from "./intent"
import { detectEmotion, fallbackEmotion, normalizeEmotion } from "./emotion"
import { analyzeRelationshipStage, buildHeuristicRelationshipStage, normalizeRelationshipStage } from "./relationship-stage"
import { buildEmotionRoute } from "./route"
import { buildReplyPolicy } from "./reply-policy"

/**
 * 对话理解链（串行，全程兜底）。
 * 顺序严格遵循文章187：intent → emotion → relationshipStage → route → replyPolicy。
 * 任一 LLM 节点失败都不会抛到主链路——内部各自 try/catch 回退 fallback。
 * 仅当整体编排异常时，外层 catch 用 fallback 重建一套完整结果。
 */
export async function analyzeConversationUnderstanding(ctx: UnderstandingContext): Promise<ConversationUnderstanding> {
  try {
    const intent = await classifyIntent(ctx)
    const emotion = await detectEmotion(ctx, intent)
    const relationshipStage = await analyzeRelationshipStage(ctx, intent, emotion)
    const route = buildEmotionRoute({ intent, emotion, relationshipStage })
    const replyPolicy = buildReplyPolicy({ safety: null, intent, emotion, relationshipStage, route })
    return { analysisVersion: "conversation-understanding-v2", safety: null, intent, emotion, relationshipStage, route, replyPolicy }
  } catch (err) {
    console.warn("conversation understanding failed, full fallback", err)
    const intent = normalizeIntent(fallbackIntent)
    const emotion = normalizeEmotion(fallbackEmotion)
    const relationshipStage = normalizeRelationshipStage(buildHeuristicRelationshipStage(ctx, intent, emotion), ctx, intent, emotion)
    const route = buildEmotionRoute({ intent, emotion, relationshipStage })
    const replyPolicy = buildReplyPolicy({ safety: null, intent, emotion, relationshipStage, route })
    return { analysisVersion: "conversation-understanding-v2", safety: null, intent, emotion, relationshipStage, route, replyPolicy }
  }
}
```

## 8. 结构化输出依赖与退路

**依赖 LLM 结构化输出的判断**（3 个，均要求合法 JSON）：
- `classifyIntent` → `ConversationIntentSchema`
- `detectEmotion` → `ConversationEmotionSchema`
- `analyzeRelationshipStage` → `ConversationRelationshipStageSchema`

**不依赖 LLM**（纯代码规则，确定性，不会因模型不稳失败）：
- `buildEmotionRoute` → `EmotionRouteSchema`
- `buildReplyPolicy` → `ReplyPolicySchema`

**若 `deepseek-v4-flash` 的 `response_format:json_object` 不稳的退路**（文章183/184/188 用 LangChain 多 method 重试，本项目无 LangChain，对应改写为三层）：

1. **Prompt 约束**：system 末尾强约束「只输出一个合法 JSON 对象，不要解释、不要 markdown 代码块」；判断类调用 `temperature:0` 提升确定性。
2. **容错提取 + Zod 解析**：`extractJson` 先 `JSON.parse`，失败则剥离 ```​```json``` 围栏、再截取首个 `{...}` 子串解析；最后 `schema.parse` 做强校验（多字段/枚举越界直接判失败→触发重试）。
3. **重试**：`callStructured` 默认重试 2 次（共 3 次）。若该模型/中转完全不支持 `response_format`，降级方案：去掉 `response_format` 字段、纯靠 Prompt 约束 + extractJson（在 catch 里第二轮自动生效，可加 `attempt>0 时不带 response_format` 的开关）。
4. **节点级兜底**：3 次仍失败 → 该节点返回 `fallback*` + `normalize*`，理解链照常产出完整结果。
5. **整链兜底**：`analyzeConversationUnderstanding` 外层 catch 重建 fallback 全集。

> 关系阶段的兜底额外带启发式（`buildHeuristicRelationshipStage` 按 messageCount 推阶段），是 5 节点里唯一「兜底也有业务逻辑」的，保证模型不可用时关系节奏仍合理（文章187）。
