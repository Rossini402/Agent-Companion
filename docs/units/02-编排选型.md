# 单元2 · 编排选型

> 精读文章：05 LangChain/LangGraph、10 LangGraph Agent 编排、21 从 Agent 开始、22 从单 Agent 到多 Agent、23 LangChain 与单 Agent、24 LangGraph 与多步骤流程
> 学习目标：理清 core / LangChain / LangGraph 三层职责；掌握单 Agent vs 多 Agent 的判断与升级时机。

---

## 核心概念：每条一句话讲清

- **Agent（智能体）**：有大脑（LLM）、记忆、工具、人设、独立行为的个体，用 `createAgent({ model, tools, systemPrompt, name })` 定义，用 `invoke()` 接任务（文章21）。
- **三层职责分工**：`@langchain/core` 定基础接口（零件），`LangChain` 把单个 Agent 组装成能工作的应用单元，`LangGraph` 把 Agent 放进更大的流程/多 Agent 系统（文章21）。
- **Agent 的五层内部结构**：输入层 → 上下文层 → 思考材料层（System Prompt/输入/检索/工具说明/历史）→ 行动层（调模型/调工具/请求外部）→ 输出层（文本/工具调用/结构化结果/下游数据）（文章21）。
- **@langchain/core 四类抽象**：消息协议（System/Human/AI/ToolMessage）统一输入输出格式、Prompt 抽象（ChatPromptTemplate/MessagesPlaceholder）让材料像组件拼装、Runnable 协议让节点用 `.pipe()` 串联、Parser/Tool 协议规范输出解析与外部能力描述（文章21）。
- **LangChain 在单 Agent 里做的六件事**：模型接入、输入组织、链路编排、工具接入、一轮请求处理、记忆与检索接入——本质是"胶水工作"（文章23）。
- **普通链 vs Agent runtime**：`prompt.pipe(model).pipe(parser)` 是你手动安排好的固定步骤；`createAgent()` 是运行时，模型可在过程中自主决定是否调工具、先调哪个、拿到结果后是否继续（文章23）。
- **单 Agent**：一个"总控台/全能助手"，自己完成意图理解、工具调度、回复生成；链路短、状态少、开发成本低、易排查（文章22）。
- **多 Agent**：把大而全的总负责人拆成职责明确的小团队（router/memory/schedule/companion），重点是**分工而非更聪明**（文章22）。
- **LangGraph 的本质**：先解决流程编排（组织流程、保存状态、处理分支与回跳），多 Agent 只是它常见的一种形态（文章24）。
- **State（共享状态/数据总线）**：所有节点读写同一个状态对象而非互相传参，把中间信息收拢成一份明确的数据契约（文章10、24）。
- **Node / Edge**：Node 是只做一件明确事的步骤，Edge 是"这步做完去哪"，把代码里隐含的流程显式化（文章24）。
- **Checkpointer（持久化/恢复）**：记住当前走到哪个节点、状态里已有哪些结果、下一步要做什么，解决断点续跑、跨会话记忆、人工确认等线上问题（文章10、24）。
- **Conditional Edges（条件路由）**：根据当前 State 决定下一个节点，路由函数**必须是纯函数**——只读不改、不调外部服务（文章10）。
- **三条路径架构**：主路径（95%+ 请求）、拦截路径（安全检查短路走兜底）、异步路径（返回用户后台跑记忆写入等），核心原则是"把用户可感知延迟与系统必须完成的工作分离"（文章10）。

## 设计决策与"为什么"

- **为什么先单 Agent**：新手最容易在需求没理清前过度拆分。链路短、状态少、开发成本低、问题易排查——早期先把"全能助手"做好通常更稳（文章22）。
- **为什么共享 State 而非节点直接传参**：实现完全解耦。节点只声明读哪些字段、写哪些字段，不需知道数据来自哪个节点，中间插入新节点不改已有代码；面向 State 字段编程即"面向接口编程"；任意节点执行完打印 State 即可调试（文章10）。
- **为什么用图建模而非 if-else**：复杂依赖与条件分支用 if-else 串联会变"意大利面"，每加模块改多处。图把隐式业务依赖变成显式拓扑，图本身就是架构文档（文章10、24）。
- **为什么记忆检索与情绪读取要并行**：当前只省 10ms，真正价值在可扩展性——未来加新模块只要不依赖并行分支输出就能并入，总延迟仍等于最慢分支。Fork/Join 由图拓扑自动调度，比手写 `Promise.all` 更易扩展（文章10）。
- **为什么分同步/异步边界**：判断依据是"该操作输出是否被当前回复所需"。需要→同步（安全检查、检索、情绪读取、Prompt 组装、LLM 生成）；只影响下次→异步（情绪更新、亲密度计算、记忆写入、摘要）。异步在 Workers 中需 `ctx.waitUntil()` 保活（文章10）。
- **为什么路由函数必须纯函数**：路由职责是"决策"而非"执行"。若路由本身要调 LLM 判断意图，应设计成独立节点而非塞进路由函数（文章10）。
- **节点粒度三原则（满足任一即拆，全不满足即合）**：独立可替换、有并行机会且延迟显著、清晰的调试边界。例：正则匹配+模型分类虽是两操作，但同属"输入安全检查"且总一起调试，故合并为一个节点（文章10）。
- **升级到多 Agent 的判断信号**：先停在单 Agent——工具 3~5 个以内、多数请求一轮内完成、状态流转不复杂、目标是先跑起来；该上多 Agent——Prompt 越来越长、工具多到模型常选错、一条请求常跨多阶段、不同任务混在一起难维护、需单独观测某角色表现（文章22）。
- **关键区分：流程变复杂 ≠ 必须多 Agent**：流程长/状态多→先理顺工作流编排、工具分组、上下文裁剪；只有当**职责明显混在一起**时才考虑拆多 Agent（文章22）。
- **查询理解是最脆弱节点**：用 LLM 输出结构化查询计划，JSON 格式错就断链路，必须有降级策略——解析失败时默认激活所有检索通道（宁多检索不漏检）（文章10）。
- **LLM 生成是延迟瓶颈（占 70~80%）**：优化其他节点收益有限，真正有效的是流式输出和模型选择（文章10）。

## 关键代码/片段

**1. 定义一个带工具与人设的单 Agent（文章21、23）**

```typescript
import { createAgent, tool } from 'langchain'
import * as z from 'zod'

const createReminder = tool(
  async ({ title, time }) => `已创建提醒：${time} ${title}`,
  {
    name: 'create_reminder',
    description: '创建提醒事项',
    schema: z.object({ title: z.string(), time: z.string() }),
  }
)

const agent = createAgent({
  model: 'openai:gpt-4.1-mini',
  tools: [createReminder, searchMemory],
  systemPrompt: '你是一个自然、克制、会照顾用户情绪的陪伴型助手。',
})

const result = await agent.invoke({
  messages: [{ role: 'user', content: '帮我记一下下周三下午两点看牙医' }],
})
```

**2. 普通链（手动编排，步骤固定）vs Agent runtime（运行时自主调度）（文章21、23）**

```typescript
// 普通链：步骤怎么串、何时查工具、何时结束，都由你写死

// Agent runtime：模型在运行中自主决定是否调工具/先调哪个/是否继续

```

**3. LangGraph 的 State 结构（数据总线，文章05）**

```typescript
import { BaseMessage } from '@langchain/core/messages'

interface AgentState {
  messages: BaseMessage[]              // 输入字段：入口写入，后续只读
  userProfile: Record<string, any>    // 用户画像
  npcMood: 'happy' | 'angry' | 'depressed' // 中间字段：某节点写，下游读
  intimacyLevel: number               // 亲密度
  longTermMemories: string[]          // 本轮检索到的长期记忆
  nextStep: string                    // 调度器决定的下一步（输出字段）
}
```

**4. 动态 Prompt 组装（LCEL，文章05）**

```typescript
const prompt = ChatPromptTemplate.fromMessages([
  ['system', '你现在的状态是 {mood}。相关回忆：{memory}。'],
  ['human', '{input}'],
])
```

**State 字段三分类契约（文章10）**：输入字段（messages，只读）、中间字段（queryPlan/memories/npcMood，某节点产、下游消费）、输出字段（response/outputSafe，末端写、调用方读）。这是架构契约而非语法约束，违反不报错但会让数据流混乱，需在注释里标明每个字段的生产者/消费者。

## 踩坑点

- **不要过早拆多 Agent**：需求没理清就拆系统是新手高频错误。多数 AI 伴侣早期需求（聊天+查记忆+写提醒+查天气）单 Agent 完全够用（文章22）。
- **"流程长"和"职责混"是两件事**：流程长先解决编排/状态/上下文裁剪，别条件反射上多 Agent；只有职责明显混在一起才拆（文章22、24）。
- **条件路由函数若写成有副作用就埋雷**：路由只能读 State，不能改 State、不能调外部服务；要调 LLM 判断意图就单独建节点（文章10）。
- **异步任务失败用户无感知**：用户已拿到回复走了，异步路径（记忆写入等）必须自己容错——幂等性（重试写两次结果一致）、最终一致性（情绪滞后一轮可接受）、死信队列（连续失败记录待人工补偿）（文章10）。
- **Serverless 异步会被中断**：CloudFlare Workers 返回响应后销毁执行上下文，未完成的异步任务被杀，必须 `ctx.waitUntil(promise)` 保活（文章10）。
- **查询理解 JSON 解析失败会断链**：用 LLM 产结构化输出的节点最脆弱，必须有降级（解析失败默认全通道检索）（文章10）。
- **节点别拆太细也别拼太粗**：太粗失去并行能力、无法单独替换、出错难定位；太细图膨胀、每条边一次状态序列化、模糊业务语义。用三原则（独立可替换/并行机会/调试边界）衡量（文章10）。
- **State 契约靠自觉**：末端节点改了 messages 不会报错，但会让数据流混乱，需文档/注释明确字段生产者与消费者（文章10）。
- **优化方向别错位**：LLM 生成占 70~80% 延迟，把记忆检索从 100ms 压到 50ms 对体验改善有限，真正该做流式输出和模型选择（文章10）。

## 与本项目(AI-Companion)的关联 + 下一步动手建议

**关联**

- 本项目就是文章一以贯之的"AI 电子伴侣"场景，六篇给出的是一条清晰的落地主线：先用 `createAgent` 搭单 Agent（聊天+记忆检索+提醒+天气），跑通后再按信号升级到 LangGraph 编排。
- 文章10 给出的 7 节点完整管线（输入安全检查 → 查询理解 → 并行【记忆检索 + 情绪读取】 → Prompt 组装 → LLM 生成 → 输出安全检查 → 异步后处理）可直接作为本项目的目标架构蓝图。
- 三层心智模型可直接套到项目分层：core 管协议、LangChain 管单 Agent 应用单元、LangGraph 管流程与多 Agent。

**下一步动手建议**

1. **先落单 Agent MVP**：用 `createAgent` 挂 3~5 个工具（searchMemory / createReminder / getWeather）+ 人设 systemPrompt，先把一轮请求跑通，不要一上来上图。
2. **定义 State Schema**：照文章05 的 `AgentState` 落地本项目状态（messages / userProfile / npcMood / intimacyLevel / longTermMemories / nextStep），并按三分类（输入/中间/输出）写明每个字段的生产者与消费者注释。
3. **接 Checkpointer**：选 SQLite/Postgres/Redis 之一，实现跨会话恢复 `npcMood` 与 `intimacyLevel`，这是"伴侣不断片"的关键。
4. **划同步/异步边界**：把情绪更新、亲密度计算、记忆写入、摘要生成放异步路径；若部署在 Workers 记得 `ctx.waitUntil`，并给异步任务加幂等。
5. **加双层安全短路**：输入侧拦截（省 LLM 调用）+ 输出侧拦截（兜底），两个条件路由点都导向同一兜底回复节点。
6. **升级触发器自检**：当出现 Prompt 越来越长 / 模型频繁选错工具 / 一条请求跨多阶段时，再把单 Agent 拆成 router + memory + schedule + companion 多 Agent，且先确认是"职责混"而非仅"流程长"。
