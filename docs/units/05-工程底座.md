# 单元5 · 工程底座

## Monorepo 与 Turborepo

> 精读范围：知识库文章 62-75（重点 62/64/65/66/71/75）。本节面向"已会基础、要落地"的工程师，把"为什么用 monorepo + 怎么用 Turborepo 把任务跑顺"提炼成可直接动手的笔记。

### 一句话总览

monorepo 是**仓库组织方式**（项目要不要放一个仓库），workspace 是**本地包连接方式**（包之间怎么互相引用），包管理器负责**装依赖**，Turborepo 负责**任务编排 + 缓存**。四层是配合关系，不是同义词。顺序是：先有仓库组织 → 再有本地包关系 → 最后才有任务编排（文章63）。

---

## 核心概念：每条一句话讲清

- **monorepo**：把原本分散在多仓库的多个项目放回同一个 git 仓库统一管理，它最先解决的不是构建速度而是**代码关系**——本该一起改的东西不再靠复制粘贴跨仓同步（文章62）。
- **monorepo ≠ 把代码堆进一个文件夹**：价值在于把边界重新摆清楚（哪些是独立应用、哪些是共享包、哪些配置该统一），是"收回一处重新整理边界"，不是取消边界（文章62）。
- **三层目录**：`apps/`（能独立启动运行的应用）+ `packages/`（不自己对外跑、被多应用复用的共享包）+ 根目录公共配置（文章65）。
- **workspace**：让包管理器把仓库里的本地目录认成包，靠"发现 → 命名 → 链接"三步，安装时把本地目录直接 link 而不去远端 registry 下载（文章66）。
- **hoisting（依赖提升）**：多个 workspace 的共同依赖被提升到根 `node_modules` 统一存放，省磁盘、加快安装，但会引出幽灵依赖（文章67）。
- **幽灵依赖（phantom dependency）**：代码用到了某个包但没在自己 `package.json` 里声明，靠 hoisting"借用"别人装的版本能跑，换环境就炸（文章67）。
- **dependsOn / `^build`**：`^` 前缀表示"先把上游依赖包的同名任务跑完再跑自己"，构建顺序交给任务图自动拓扑排序，不靠人脑记（文章71）。
- **outputs**：声明任务真正产出的产物范围，缓存命中时据此恢复产物；写不准会出现"cache hit 但 dist 为空"（文章71/73）。
- **inputs**：声明哪些文件变化才算任务"真的变了"；不配则默认所有 git 跟踪文件都纳入，缓存命中率被拖低（文章72）。
- **env / globalEnv**：声明哪些环境变量参与缓存 key 计算；漏声明会导致"缓存命中但产物里是旧值/别环境的值"（文章72/74）。
- **本地缓存 vs 远程缓存**：本地缓存让同一台机器别重复算；远程缓存把结果放到团队共享处，让同事和 CI 在相同输入下直接复用（文章73）。
- **`--filter`**：限定只跑受影响的包；`--filter="...[origin/main]"` 找出相对主分支有变化的包及其下游消费方（CI 最常用）（文章72/74）。

## 设计决策与"为什么"

- **AI 应用为什么更早走到 monorepo**：它天然长出多个强相关层（web/admin/api/worker + 共享 schema/prompts/ui/ai-core/评测脚本），且最容易联动的是**上下游约定**（请求参数、响应结构、任务状态枚举、工具调用结果格式、评测输出格式），一改全改。分仓后这本是一件事却变成多次提交、多次发版、多次联调（文章64）。但项目还小、几乎没有共享层时，保持简单更好——monorepo 的价值信号是"关系开始变紧"而非"听起来很大"（文章64）。
- **第一刀为什么切在 apps / packages 之间**：提前隔离最易打架的边界。不切会出现两种烂法：业务代码被"顺手共享"出去污染 packages，或共享代码无处安放被各应用各复制一份。规则简单：**应用代码先待在应用里，只有真正跨应用复用的才往 packages 提**（文章65）。
- **不要一次把未来目录全建出来**：项目还小就分出一堆空包只会让结构先复杂。先放真正已存在的边界，按需长（文章65）。
- **共享代码"晚一点但更准确"**：能复用 ≠ 该立刻共享。太早抽，packages 会堆满只服务单个应用的业务细节，反而把边界搞乱。更稳的顺序是先留 apps，等它"被多处稳定使用 / 命名不再绑定具体页面 / 输入输出固定 / 变化像公共能力演进"再提（文章68）。判断标准升级版：共享后别人对它的预期会不会变错——packages 里的东西会被默认当成稳定公共能力（文章68）。
- **内部包要不要 build——核心只有一条：谁来把 TS 编译成 JS**（文章69）：
  - 策略一 源码直接导出：由消费方（Next.js `transpilePackages` / Vite）编译；改动即时生效、HMR 正常、无额外 build；不适合发 npm。
  - 策略二 构建成 dist 产物：包自己用 tsup/tsc 编译；边界清晰利于 Turborepo 增量缓存、适合发 npm / 消费方是 Node 服务；代价是改完要先 build，且要在 turbo 里声明 `ui#build` 是 `web#dev` 前置。
  - 策略三 tsconfig paths 映射：配置简单，但只有 TS 懂，运行时需各自配别名，包多了维护成本高。
  - 选择：会发 npm → 策略二；只内部用 + 消费方 Next/Vite → 策略一最省事；消费方是 Node 服务 → 策略二（或加 tsx/ts-node）。**不同包混用不同策略完全正常**（文章69）。
- **多运行环境的边界由"运行时"决定而非"能不能 import"**：浏览器/Node/Worker 代码不能因共享方便混在一起。按环境拆包（`shared` 纯环境无关 / `shared-node` / `shared-browser`），用 conditional exports + `engines` 给出信号（文章70）。
- **tsconfig 用 extends 基线分层**：根/`packages/config` 放 `tsconfig.base.json`，再派生 `tsconfig.nextjs.json`、`tsconfig.node.json`，各 app extends 对应基线——好处是全仓升级某个 TS 选项只改一处（文章70）。
- **dev 和 build 写法不同的原因**：build 是"算出产物就结束"的任务（配 outputs、可缓存）；dev 是持续挂着的长进程（`cache:false` + `persistent:true`）。是否缓存取决于任务在"产出结果"还是"持续服务"（文章71）。
- **远程缓存不是高级附加功能**，而是把"结果复用"从个人扩展到团队；真正好用的标志是团队**信任**缓存——命中不是碰巧没跑，重算不是系统抽风（文章73）。
- **CI 里要专门处理 monorepo**：CI 每次是全新环境、无本地 node_modules 和缓存，不配置就退化成全量构建。两件事：接远程缓存（降 CI 构建时间 50%-80%）+ 管好环境变量分层（文章74）。

## 关键代码/片段

**1. 三层把本地包连起来（workspace，文章66）**

```yaml
# pnpm-workspace.yaml —— 第一层：声明 workspace 范围
packages:
  - 'apps/*'
  - 'packages/*'
```

```jsonc
// packages/shared/package.json —— 第二层：共享包声明包名
{ "name": "@acme/shared", "version": "0.0.0", "main": "./src/index.ts" }

// apps/web/package.json —— 第三层：应用把共享包写进依赖
{ "name": "@acme/web", "dependencies": { "@acme/shared": "workspace:*" } }
```

**2. 单例包锁版本，内部包用 peerDependencies（文章67）**

```jsonc
// 根 package.json —— 强制 react 单例（yarn 用 resolutions / npm·pnpm 用 overrides）
{ "resolutions": { "react": "19.1.0", "react-dom": "19.1.0" } }

// packages/ui/package.json —— 不自带 react，由使用方提供，避免双实例
{ "name": "@acme/ui", "peerDependencies": { "react": ">=18.0.0" } }
```

**3. 三层任务结构 + 最小 turbo.json（文章71）**

```jsonc
// 根 package.json —— 收成统一入口
{ "name": "acme-monorepo", "private": true,
  "scripts": { "dev": "turbo run dev", "build": "turbo run build", "lint": "turbo run lint" },
  "devDependencies": { "turbo": "^2.0.0" } }
```

```jsonc
// turbo.json
{ "$schema": "https://turbo.build/schema.json",
  "tasks": {
    "build": { "dependsOn": ["^build"], "outputs": ["dist/**", ".next/**", "!.next/cache/**"] },
    "dev":   { "cache": false, "persistent": true },
    "lint":  { "outputs": [] }
  } }
```

**4. 让缓存"真正可信"的进阶字段（文章72）**

```jsonc
{ "$schema": "https://turbo.build/schema.json",
  "globalDependencies": ["tsconfig.json"],
  "globalEnv": ["NODE_ENV"],
  "tasks": {
    "build": {
      "dependsOn": ["^build"],
      "inputs": ["src/**", "package.json", "tsconfig.json"],
      "outputs": ["dist/**", ".next/**", "!.next/cache/**"],
      "env": ["NEXT_PUBLIC_API_URL", "NEXT_PUBLIC_APP_ENV"]   // 凡打包进产物的 env 必须列
    },
    "test": { "dependsOn": ["build"], "inputs": ["src/**", "test/**", "vitest.config.ts"], "outputs": ["coverage/**"] }
  } }
```

**5. per-package 覆盖产物路径（不同 app 产物不同，文章70）**

```jsonc
// apps/web/package.json —— web 产 .next，单独覆盖 outputs
{ "name": "@acme/web", "turbo": { "build": { "outputs": [".next/**", "!.next/cache/**"] } } }
```

**6. Next.js 内部包源码直出（策略一，文章69）**

```ts
// apps/web/next.config.ts
import type { NextConfig } from 'next'
const nextConfig: NextConfig = { transpilePackages: ['@acme/ui', '@acme/shared'] }
export default nextConfig
```

**7. 常用 filter / 调试命令（文章72/74）**

```bash
turbo run build --filter=@acme/web              # 只构建该包
turbo run build --filter=@acme/web...           # 该包 + 所有上游依赖
turbo run build --filter="...[origin/main]"     # 相对 main 有变化的包及其下游（CI 常用）
turbo run build --dry                           # 只看任务图，排查"为啥这个包被拉进来/循环依赖"
turbo run build --summarize                     # 生成 .turbo/runs/ 报告，看 hash 怎么算、为啥没命中
```

**8. CI 接远程缓存（GitHub Actions，文章74）**

```yaml
# .github/workflows/ci.yml
jobs:
  build:
    runs-on: ubuntu-latest
    env:
      TURBO_TOKEN: ${{ secrets.TURBO_TOKEN }}   # 走 Secrets，不明文写
      TURBO_TEAM:  ${{ secrets.TURBO_TEAM }}
    steps:
      - uses: actions/checkout@v4
        with: { fetch-depth: 2 }                # 关键：浅克隆会让 diff 失效退化成全量构建
      - uses: actions/setup-node@v4
        with: { node-version: 20, cache: 'yarn' }
      - run: yarn install --frozen-lockfile     # 锁版本保证可重复
      - run: turbo run build lint test
```

## 踩坑点

- **幽灵依赖**：用到的包必须自己声明，别借 hoisting 提升上来的版本；在"别人升级了该包 / 该 app 被移走 / CI 只装部分 workspace"时本地隐形、线上炸。pnpm 默认严格链接能从根上杜绝；yarn/npm 宽松，需自己警惕（文章67）。
- **单例包多版本共存**：react / react-dom 这类必须单例，两版本并存会让组件树跑着两个 React，行为不可预测；用 resolutions/overrides 锁版本（文章67）。
- **outputs 写不准 → "cache hit 但 dist 为空"**：缓存命中时无法正确恢复产物；不同 app 产物路径不同（`.next/` vs `dist/`）时务必按包覆盖 outputs（文章71/70/73）。
- **env 漏声明 → 缓存污染**：`NEXT_PUBLIC_*` / `VITE_*` 这类构建期打进产物的变量没进 `env`，会出现 staging 缓存被 production 命中、产物里仍是错误 API 地址（文章72/74）。
- **inputs 太宽 → 缓存白配**：不配默认全 git 文件，改 README、改 CI yaml 都让 build 缓存失效；判断标准"该文件变了结果不会变就别放 inputs"。但 globalDependencies/globalEnv 也别堆太多，否则全局失效让缓存形同虚设（文章72）。
- **循环依赖**：`packages/shared` 与 `packages/ui` 互相引用 → `^build` 死锁无法拓扑排序、TS 解析不可预测。用 `turbo run build --dry` 看任务图发现；处理是把互引部分提到第三个包，或砍掉本不该有的引用（文章75）。
- **运行环境边界变模糊**：浏览器代码依赖了 `node:fs`、worker 专属逻辑被引进前端——TS 不拦、构建不拦，只有运行时炸且难追溯。按运行环境拆共享包（文章70/75）。
- **策略二忘了声明 build 前置**：内部包走 dist 产物却没在 turbo 里声明 `ui#build` 是 `web#dev` 前置，启动顺序会错（文章69）。
- **CI 浅克隆**：`fetch-depth` 默认浅克隆让 Turborepo 无法 git diff，退化成全量构建，至少取 2 个提交（文章74）。
- **lockfile 冲突日常化**：多人多分支同时改依赖，`pnpm-lock.yaml` 这种大扁平文件难自动合并。缓解：集中处理依赖升级、冲突时删 lockfile 重新 install（别手改）、定期 dedupe（文章75）。
- **"反正都在一个仓库里"的松弛感**：边界全靠"以后再收"，依赖图越来越乱、某些包被半个仓库引用却说不清职责。判断信号：**改一处越来越难说清影响范围，就是某条边界该重审了**（文章75）。

## 与本项目(AI-Companion)的关联 + 下一步动手建议

**关联**：本项目正是文章64描述的典型 AI 陪伴 Agent 形态——会同时长出 web（陪伴前端）、api（接口层）、worker（异步 AI 任务/编排）、以及共享 schema、prompts、ai-core（模型调用封装）、评测脚本。这些层的**上下游约定**（请求参数、响应结构、任务状态枚举、工具调用结果格式、评测输出格式）天然一改全改，分仓维护成本极高，是 monorepo 的标准适配场景。

**下一步动手建议（按优先级）**：

1. **先搭最小三层骨架**：`apps/web` + `apps/api` + `packages/shared`（纯类型/工具）+ 根目录 `pnpm-workspace.yaml` / `package.json` / `turbo.json`。包管理器优先 pnpm（默认严格链接，从根上杜绝幽灵依赖）。不要一次把 admin/docs/worker/ai-core 全建空。
2. **把上下游约定优先沉到 `packages/schemas`**：陪伴 Agent 的任务状态枚举、请求/响应结构、function-calling 工具定义是最先"一改全改"的，是最值得早抽的共享层；prompts 模板、ai-core 模型封装等长稳定后再提。
3. **严守运行环境边界**：模型调用/`fs`/密钥相关放 `shared-node`，前端 DOM/storage 相关放 `shared-browser`，`shared` 只留环境无关纯逻辑——避免把 Node-only 的模型调用代码被前端顺手 import 而运行时炸。
4. **turbo.json 一开始就配准 inputs/outputs/env**：尤其把所有 `NEXT_PUBLIC_*` / 模型 API 地址类构建期变量列进 `env`，防止 staging/production 缓存污染；dev 任务 `cache:false + persistent:true`。
5. **CI 接 Vercel 远程缓存 + `--filter="...[origin/main]"`**：陪伴项目迭代频繁，PR 只动 prompts 或某个 app 时只构建受影响包，配合 `fetch-depth:2` 和 `--frozen-lockfile`。
6. **内部包默认走策略一（源码直出 + Next transpilePackages）**：减少每天 build 摩擦；只有要对外发布或被纯 Node 服务消费的包（如独立部署的 worker）才转策略二。
7. **建立边界自检习惯**：定期 `turbo run build --dry` 看任务图，发现循环依赖或"被半个仓库引用却职责不清"的包就及时拆——把文章75那句判断标准（改一处难说清影响范围=该重审边界）写进团队规范。


## Hono + Cloudflare

> 精读范围：文章 76-101（重点 78-85 基础链路 / 86-89 存储 / 91 流式 / 96 RAG / 97 DO+WS）。本子主题对应工程底座的「API 框架 + 边缘运行时」一层，是 AI-Companion 后端落地的实际技术栈。

### ## 核心概念：每条一句话讲清

- **Hono**：基于 Web Standards（`Request`/`Response`/`fetch`）的轻量框架（gzip ~14KB、零依赖），处理函数本质就是「收 Request，返 Response」，因此天然跨运行时（文章76）。
- **Express vs Hono 的根本差异**：Express 绑死 Node.js 的 `IncomingMessage`/`ServerResponse`，Hono 站在标准对象之上，所以同一段代码能跑在 Workers/Deno/Bun/Node（文章76）。
- **Cloudflare Workers**：把「代码」而非「数据」部署到全球边缘节点，用 V8 isolate（非容器/VM）实现 <5ms 冷启动（文章77）。
- **V8 isolate**：比容器更轻的 JS 沙箱，多个 isolate 共享一个 V8 进程、内存互相隔离，无需启动完整运行时（文章77）。
- **Workers 的硬约束**：免费版每请求 10ms CPU（付费默认 30s）、内存 128MB、无文件系统、非完整 Node.js——定位是「快进快出，不做重计算」（文章77）。
- **Bindings**：Workers 通过 `c.env.XXX` 访问 KV/D1/R2/AI 等服务，免连接字符串和认证，配置写在 `wrangler.jsonc`（文章77、92）。
- **Context（`c`）**：单一入口对象，`c.req` 读请求、`c.json/text/html` 写响应、`c.set/get` 在中间件↔路由间传数据（作用域为单次请求）（文章80）。
- **洋葱模型中间件**：`await next()` 把中间件切两半，`next()` 前处理请求、后处理响应；先注册的最先接触请求、最后接触返回（文章81）。
- **数据校验（zValidator）**：用一份 Zod schema 同时完成「运行时校验 + 编译期类型推导」，`c.req.valid('json')` 自动带类型（文章82）。
- **错误处理三件套**：`HTTPException` 抛业务错误、`app.onError` 统一捕获、`app.notFound` 兜底 404；约定「能正常返回用 return，异常中断用 throw」（文章83）。
- **认证（Authentication）= 你是谁**：Workers 无状态，Session 表无处可存，故用 JWT——请求自带签名身份信息，服务端验签即可，不查库（文章84）。
- **JWT 是签名不是加密**：Payload 仅 Base64 编码（任何人可解码），签名只防篡改；不要往里放敏感信息（文章84）。
- **鉴权（Authorization）= 你能干什么**：在 `jwt()` 之后加 `requireRole`/`requirePermission` 中间件，从 payload 取 role/permissions 判断（文章85）。
- **KV**：全球分布式键值存储、最终一致、读多写少，只能 put/get/delete/list，用 key 前缀模拟分类（文章86）。
- **D1**：边缘 SQLite 关系库，支持标准 SQL、迁移文件管理表结构，适合读多写少的结构化数据（文章87）。
- **Drizzle ORM**：TypeScript 优先、SQL-like 语法的 ORM，schema 即类型，解决原生 SQL 无类型/拼字符串/表结构散落三大痛点（文章88）。
- **R2**：S3 兼容对象存储，存文件（头像/附件/AI 图片），最大卖点是零出口流量费（文章89）。
- **Hono RPC（`hc`）**：后端链式定义路由 + `export type AppType`，前端 `hc<AppType>` 拿到全链路类型推导的客户端，底层仍是 fetch（文章90）。
- **流式响应**：边生成边发，AI 逐 token 输出的基础；Hono 三种 API——`streamText`/`stream`/`streamSSE`（文章91）。
- **SSE**：`Content-Type: text/event-stream` 的结构化事件流（event/id/data），单向、走普通 HTTP、浏览器原生自动重连，是 LLM 流式接口的事实标准（文章91）。
- **项目结构**：按职责拆 `routes/middleware/db/lib`，入口只负责组装；类型集中到 `types.ts` 的 `AppEnv`（文章92）。
- **Workers AI**：`c.env.AI.run(modelId, input)` 在边缘跑开源模型（Llama/bge/whisper/flux），低延迟、有免费额度（文章95）。
- **AI Gateway**：给第三方 LLM 调用加代理层，Dashboard 配缓存/限流/重试/日志/语义缓存，改 `baseURL` 即可接入（文章95）。
- **Vectorize**：Cloudflare 向量数据库，配合 Workers AI 的 embedding 实现 RAG 的「检索」环节（文章96）。
- **Durable Objects（DO）**：「有身份的 Worker」，`idFromName` 让同名永远路由到同一实例，含内存状态 + 强一致存储 + 可休眠 WebSocket，是有状态场景（会话记忆/长连接/限流）的解法（文章97）。
- **Queues / Cron**：Cron 按时间触发 `scheduled()`，Queue 是生产者-消费者异步队列（at-least-once、重试、DLQ）；常组合用（Cron 切片塞队列）（文章98）。
- **四层缓存**：响应头 → Cache API（节点级）→ Hono cache 中间件 → KV（全球一致）；AI 项目重点缓存 embedding、确定性 LLM 响应、语义缓存（文章99）。
- **可观测性**：Workers 日志默认不持久化，开 `observability.logs.enabled` 即可在 Dashboard 查；结构化 JSON 日志 + requestId + Analytics Engine + Sentry（文章100）。

### ## 设计决策与"为什么"

- **为什么选 Hono 而非 Express**：边缘环境里包体积和冷启动是「入场资格」而非优化项；Express 依赖链 >1MB 且绑 Node 模块，塞不进 Workers（文章76、77）。
- **为什么 Workers 无状态 → 必须用 JWT**：每次请求是全新 isolate，内存无 Session 表；存 KV/DB 又多一次查询延迟。JWT 让「光看请求就知道你是谁」（文章84）。
- **为什么 `secret` 用环境变量 + `wrangler secret`**：硬编码密钥会进 Git；非敏感配置写 `vars`（可提交），密钥用 `wrangler secret put`（加密存云端），本地用 `.dev.vars`（必须 gitignore）（文章84、92）。
- **为什么校验和类型用同一份 Zod schema**：传统做法手写 interface + 另写校验，两份易不同步出 bug；`z.infer` 让 schema 成为单一真相源（SSOT）（文章82、101）。
- **为什么 CORS 要在鉴权之前**：浏览器跨域先发 OPTIONS 预检（不带 token），若鉴权在前会被 401 拦掉。推荐顺序 logger→cors→secureHeaders→鉴权→业务（文章81）。
- **为什么「业务失败」统一用 throw**：`return c.json` 不进 `onError`，`throw` 才会；统一 throw 能把所有错误汇总到一处，路由只管「数据合法后写业务」（文章83）。
- **为什么 ValidationError 的 instanceof 判断要放 HTTPException 前面**：前者继承后者，反过来写会被父类先匹配（文章83）。
- **KV vs D1 怎么选**：能设计稳定 key 且主要按 key 读 → KV；开始想「按时间排序/条件筛选/查最近 N 条」→ 上 D1（文章86）。
- **为什么选 Drizzle 而非 Prisma**：Prisma 有独立 schema 语言、运行时重，边缘环境不舒服；Drizzle 是纯 TS、SQL-like、原生支持 D1、零依赖轻量（文章88）。
- **为什么 Drizzle 实例要每请求新建**：Workers 无状态、无持久连接池，`drizzle(c.env.DB)` 只是轻量包装，不能放顶层（文章88）。
- **为什么 D1 复杂事务建议用 batch 而非 transaction**：D1 驱动下 Drizzle 的 `transaction()` 受限，`batch()` 在单个事务里执行多条 SQL 更稳（文章88）。
- **R2 预签名 URL 的取舍**：想让文件流量不经过 Worker（省延迟/带宽）就用预签名 URL，客户端直连 R2（文章89）。
- **为什么 RPC 必须链式写法**：`app.get()` 返回带「已注册路由」类型信息的新类型，分开写 TS 推不出路由信息；`app.route()` 挂载时返回值也要接住再 `typeof` 导出（文章90）。
- **EventSource vs fetch 消费 SSE**：`EventSource` 自动重连但只支持 GET；AI 聊天要 POST 发消息体，必须用 `fetch + reader` 手动解析 SSE（文章91）。
- **为什么 `waitUntil` 记录用量**：响应一返回 Worker 就结束；`c.executionCtx.waitUntil(promise)` 让「响应先发回、后台跑完日志再关」，不拖慢响应（文章94）。
- **为什么流式代理要 buffer 拼接**：网络一次 `read()` 可能拿到半行/两行半，需缓冲区按 `\n` 切分、最后不完整的留到下一轮（文章91、94）。
- **Workers AI vs AI Gateway**：跑开源模型/成本敏感/隐私 → Workers AI；要 GPT-4o/Claude Opus 顶级模型 + 缓存限流日志 → 第三方 SDK + AI Gateway；常两者并用（核心对话走 Gateway，embedding/分类走 Workers AI）（文章95）。
- **为什么多租户用 namespace 而非 metadata 过滤**：metadata 是查询时软过滤（代码忘加条件就泄露），namespace 是物理隔离更安全（文章96）。
- **何时用 DO / 何时不用**：有自然分片键（房间/用户/会话 ID）→ 用 DO；全局单例计数器/任意缓存/大库 → 分别用 Analytics Engine/KV/D1。DO 新建一律用 SQLite-backed（更便宜、可 SQL）（文章97）。
- **为什么 WebSocket 用 Hibernatable API**：传统 `accept()` 连接期 DO 必须常驻内存按时计费；`acceptWebSocket()` 空闲休眠不计费，对「挂着聊天页但不说话」省大量成本（文章97）。
- **Cron vs Queue**：定时/周期/无状态/失败等下次 → Cron；用户请求触发/有状态/要重试 DLQ → Queue。数据量大时 Cron 扫全表会超时（单次 30s CPU），改为 Cron 切片塞 Queue（文章98）。
- **为什么消费者必须幂等**：Queue 是 at-least-once，同消息可能投递多次；消息体带业务唯一 ID，消费前查「处理过了吗」（文章98）。
- **缓存「public + max-age」是泄露用户数据的常见配方**：`/users/me` 这类绝不能 public，否则 CDN 缓存后别人看到你的数据，必须 `private`（文章99）。
- **Next.js 集成模式选择**：要 Workers binding / 多端 API / 中大团队 → 模式 A（独立部署 + RPC）；单体/小团队/无多端 → 模式 B（嵌入 Route Handler）。可先 B 起步再拆 A（文章101）。

SSE 转发 LLM 流（AI 应用核心）：

```typescript
import { streamSSE } from 'hono/streaming'
app.post('/chat', async (c) => {
  const { messages } = await c.req.json()
  const upstream = await fetch('https://api.openai.com/v1/chat/completions', {
    method: 'POST',
    headers: { Authorization: `Bearer ${c.env.OPENAI_API_KEY}`, 'Content-Type': 'application/json' },
    body: JSON.stringify({ model: 'gpt-4o', messages, stream: true }),
  })
  return streamSSE(c, async (stream) => {
    const reader = upstream.body!.getReader()
    const decoder = new TextDecoder()
    let buffer = ''
    while (true) {
      const { done, value } = await reader.read()
      if (done) break
      buffer += decoder.decode(value, { stream: true })
      const lines = buffer.split('\n'); buffer = lines.pop() || ''   // 半行留到下一轮
      for (const line of lines) {
        if (!line.startsWith('data: ')) continue
        const data = line.slice(6)
        if (data === '[DONE]') return
        const content = JSON.parse(data).choices[0]?.delta?.content
        if (content) await stream.writeSSE({ data: content, event: 'message' })
      }
    }
  })
})
```

### ## 踩坑点

- **请求体只能读一次**：`c.req.json()` / `c.req.text()` / `c.req.formData()` 三选一，消费后不能再读（文章80）。
- **路由顺序**：越具体越靠前，兜底/通配符靠后；`/users/me` 必须在 `/users/:id` 前面，否则 `me` 被当 id，`Number('me')=NaN`（文章79、93）。
- **app.route 挂载顺序**：先在子模块写完所有路由，再 `app.route()` 挂载；反过来「先挂载再追加」会 404（文章79）。
- **路径/查询参数都是字符串**：需要数字自己转，或用 `z.coerce.number()`（文章79、82）。
- **JSON 校验需正确 Content-Type**：缺 `application/json` 服务端不按 JSON 解析，会出现「传了数据却校验不过」（文章82）。
- **KV `list()` 只返回 key 信息不含 value**：要值得对每个 key 再 `get()`；KV 最终一致，刚写入不保证立即全球可读（文章86）。
- **D1 限制**：免费单库 500MB；写延迟高于读；不支持事务嵌套；SQLite 语法非 MySQL/PG；单查最多返 5MB（文章87）。
- **Drizzle `db.query` 关联查询需初始化时传 `{ schema }`**；`relations()` 不生成外键，外键靠 `.references()`（文章88）。
- **R2 key 是扁平字符串**：`uploads/2024/x.png` 不存在真实文件夹层级；单 PUT 最大 5GB（超过用 multipart）（文章89）。
- **RPC 三大硬约束**：必须链式写法、前后端需共享类型（monorepo）、类型只在编译期生效（运行时版本不一致照样出错）（文章90）。
- **EventSource 只支持 GET**：POST 流式场景用 `fetch + reader` 手动解析（文章91）。
- **密码哈希**：Workers 不支持 bcrypt，示例用 SHA-256 加盐仅为教学；生产应用 PBKDF2（`crypto.subtle.deriveBits`）（文章93）。
- **登录错误提示要统一**：邮箱不存在和密码错误都返回「邮箱或密码错误」，避免攻击者枚举已注册邮箱（文章93）。
- **Cron 用 UTC 时间**：中国时区写定时任务记得减 8 小时；`wrangler dev` 下 Cron 不自动触发，用 `/__scheduled?cron=...` 手动触发（文章98）。
- **Vectorize 写入是异步的**：`upsert` 返回 `mutationId`，真正可查常要几秒，实时性高的场景注意（文章96）。
- **Workers Observability 错误日志不受采样影响**：5xx 会额外全量记录；别把所有 `console.error` 都 `captureException`（Sentry 按事件计费）（文章100）。
- **Next.js 模式 B 必须 `basePath('/api')` + 每个 HTTP 方法单独 `export const GET = handle(app)`**；模式 A 必须显式配 CORS 且 credentials 时 origin 不能用 `*`（文章101）。

## Zod 类型校验

> 精读范围：文章 102-115（重点 103/111/113/114/115）。本子主题讲的是「用一份 Schema 同时拿到运行时校验 + 静态类型」，以及把它贯穿前端表单 → Hono 接口 → LLM 输出 → 响应的全链路工程实践。

### ## 核心概念：每条一句话讲清

- **TypeScript 类型在运行时不存在**：编译后类型信息被全部抹掉，任何「从外部世界进来的数据」（HTTP/localStorage/第三方 API/LLM JSON）都需要一个运行时守门员，这就是 Zod 的位置。（文章102）
- **Zod = Schema 校验库**：一份 Schema 同时完成「运行时校验」和「静态类型推导」，`z.infer<typeof S>` 从 Schema 反推 TS 类型，免去「手写 interface + 手写校验」的重复。（文章102）
- **单一真实来源（SSOT）**：同一份数据结构只允许存在一份权威定义，TS 类型/后端校验/前端表单/接口文档/LLM 工具定义全部从它派生，避免「模型分裂（Model Drift）」。（文章103）
- **parse vs safeParse**：`parse` 合法返回数据、不合法抛 `ZodError`；`safeParse` 永不抛错，返回可辨识联合 `{success:true,data} | {success:false,error}`。判据：失败是异常 → parse；失败是业务分支 → safeParse。（文章104）
- **ZodError.issues**：错误核心数组，每条带 `path`（出错字段路径）/`code`/`message`；`flatten()` 适合扁平表单、`format()` 适合嵌套结构。（文章104）
- **Zod 默认严格**：不做隐式类型转换、不容忍、不猜测（`z.number()` 拒绝 `"42"`，`z.boolean()` 拒绝 `1`/`'true'`，`z.date()` 只吃 Date 对象），要宽松必须显式用 `z.coerce.*` 或 `transform`。（文章105）
- **object + array 是 Schema 的脊梁**：真实项目 80% 的 Schema 是这两者嵌套组合；`z.record()` 表示 `Record<string,T>`。（文章106）
- **5 种「没值」状态**：`.optional()`(允许 undefined) / `.nullable()`(允许 null) / `.nullish()`(两者都允许) / `.default(v)`(仅 undefined 时填默认值) / `.catch(v)`(校验失败时兜底)。（文章107）
- **多选一的 5 个工具**：`literal` / `enum` / `nativeEnum` / `union` / `discriminatedUnion`；看到「几种对象形态之一且有共享辨识字段」反射弧应是 `discriminatedUnion`。（文章108）
- **refine / superRefine 描述「规则」而非「形状」**：跨字段校验、条件必填、与外部知识比对等业务规则；refine 在内置校验通过后才运行。（文章109）
- **transform / preprocess / pipe**：Zod 不只校验还能加工——`transform` 校验后变形、`preprocess` 校验前预处理、`pipe` 把上一段输出接给下一段。（文章110）
- **input ≠ output**：Schema 本质是 `input→output` 的函数；一旦出现 `default/transform/coerce/catch/preprocess/pipe`，`z.input<T>`（parse 前）和 `z.output<T>`（=`z.infer`，parse 后）就会分叉。（文章111）
- **Schema 组合派生**：`.extend/.merge/.pick/.omit/.partial/.required/.deepPartial/.and` 让一份主 Schema 派生出整套相关 Schema（公开视图/注册体/更新体/登录体）。（文章112）
- **Zod + Hono**：`zValidator('<target>', schema)` 校验 body/query/param/header/form，出口也用派生 Schema `parse` 防字段漏与敏感泄露，配合 Hono RPC client 实现端到端类型推导。（文章113）
- **Zod + LLM**：LLM 的 JSON 永远不守规矩，Zod 是把「概率性组件」接回「确定性系统」的桥；`describe()` 让一份 Schema 同时充当 prompt 指令。（文章114）

### ## 设计决策与"为什么"

- **为什么不用手写 interface + if 校验**：两份会各自走形（Model Drift），「重复 + 容易走形」才是 bug 源头。Schema 当唯一权威，改一处全链路同步（年龄上限 120→150 只改一行）。（文章103）
- **为什么默认 strip 未知字段**：`z.object` 默认丢掉未知字段（不是报错也不是保留）。接外部 API/LLM 用默认 strip，版本化严格契约用 `.strict()`，网关透传用 `.passthrough()`。坑：被丢掉的字段不会进日志。（文章106）
- **为什么安全场景优先 .pick() 而非 .omit()**：主 Schema 之后加字段时，`.omit()` 黑名单会把新字段默默带出去造成泄露，`.pick()` 白名单不会。这是安全考量不是写法偏好。（文章112、113）
- **为什么 input/output 要分别导出给前后端**：前端构造请求体用 `z.input`（可省略默认字段、可传字符串），后端 handler 用 `z.output`/`z.infer`（已填默认值、已 coerce）。用反了不会有运行时 bug，但逼你写一堆无意义的 `?.` 和非空断言。（文章111）
- **为什么响应也要校验（防御自己的后端）**：出口 `parse` 在开发期抓三类问题——DB 字段改了忘改响应、业务层误返敏感字段、外部/LLM 返回结构不一致。「请求校验防用户乱传，响应校验防自己乱写」。（文章113）
- **为什么 discriminatedUnion 优于 union**：union 串行依次尝试，性能差且失败时把所有分支错误混在一起；discriminatedUnion 读一次辨识字段直接跳分支，错误精准、类型推导也能收窄。（文章108）
- **为什么 transform 不该做校验**：transform 是「加工」不是「质检」，应保持纯函数；校验用 refine/内置方法，I/O 副作用放业务层（别在 transform 里查数据库）。（文章110）
- **为什么 LLM 输出失败是业务分支而非异常**：模型脏输出是家常便饭，必须用 safeParse + 重试（错误塞回 prompt 让模型修正）+ 降级（`.catch()` 字段兜底）+ 日志（必带 raw，否则没法排查模型漂移）。（文章114）
- **为什么 Schema 要导出本体而不只导类型**：只导类型则别的模块无法 parse，Hono RPC 的类型推导链也会断；Schema 本体可复用于校验、派生子 Schema、生成文档。（文章111、113）
- **同一份 LLM 工具 Schema 用三次**：转 JSON Schema 喂模型 API、校验模型返回的工具参数、给 TS 推导类型——SSOT 在 AI 场景的终极形态。（文章103、114）

**LLM 脏输出防御层（文章110、114、115）——可直接复用的工厂**
```ts
const stripFence = (s: string) =>
  s.replace(/^```(?:json)?\s*/i, '').replace(/\s*```$/i, '').trim()
const extractJson = (s: string) => {
  const c = stripFence(s)
  const first = c.indexOf('{'), last = c.lastIndexOf('}')
  if (first === -1 || last === -1) throw new Error('no JSON object found')
  return c.slice(first, last + 1)
}
export const LLMJsonSchema = <T extends z.ZodTypeAny>(shape: T) =>
  z.string()
    .transform(extractJson)
    .transform(s => JSON.parse(s))
    .pipe(shape) // 关键：变形后继续走严格校验
```

**Hono 统一校验封装 + 全局 ZodError 兜底（文章113）**
```ts
import { zValidator as zv } from '@hono/zod-validator'
export const validate = <T extends ZodSchema>(target: keyof ValidationTargets, schema: T) =>
  zv(target, schema, (result, c) => {
    if (!result.success) {
      return c.json({
        ok: false, code: 'VALIDATION_ERROR',
        errors: result.error.issues.map(i => ({ field: i.path.join('.'), message: i.message })),
      }, 400)
    }
  })

app.onError((err, c) => {
  if (err instanceof ZodError) {
    return c.json({ ok:false, code:'VALIDATION_ERROR',
      errors: err.issues.map(i => ({ field: i.path.join('.'), message: i.message })) }, 400)
  }
  return c.json({ ok:false, code:'INTERNAL', message: err.message }, 500)
})
```

### ## 踩坑点

- **z.number() 不吃数字字符串**：query 参数 `?page=2` 里的 `2` 是字符串，必须 `z.coerce.number()`，这是「最常见的 Hono+Zod bug」。（文章105、113）
- **z.coerce.boolean('false') 返回 true**：因为 `Boolean('false')` 为真，环境变量场景尤其坑，用 `z.enum(['true','false']).transform(v=>v==='true')`。（文章105）
- **z.date() 只接受 Date 对象**，拒绝字符串和时间戳；HTTP 场景应改用 `z.string().datetime()` 或 `z.coerce.date()`。（文章105）
- **object 默认 strip 静默丢字段**：未知字段不会报错也不进日志，排查时容易「怀疑人生」。（文章106）
- **空数组默认合法**：`z.array(...)` 允许 `[]`，要禁止用 `.min(1)`；`.nonempty()` 会把类型变成元组 `[T,...T[]]`，只在真需要时用。（文章106）
- **.optional() 加在哪里语义不同**：加在外层 = 整个对象可不存在；加在字段 = 字段可不填。（文章106）
- **.default() 只处理 undefined，不处理 null**：传 null 不会触发默认值反而报错；动态默认值（时间/UUID）必须传函数否则只在 Schema 创建时算一次。（文章107）
- **.catch() 会吞掉所有错误**：最危险的方法，让数据错误悄无声息被吃掉，必须配日志。`.default` 处理缺省、`.catch` 处理校验失败，差一字差很多。（文章107）
- **z.enum 从数组创建要加 as const**：否则 TS 推成 `string[]`，拿不到精确字面量。（文章108）
- **.refine() 只在前置校验通过后运行**：函数参数类型已被收窄，可放心当正确类型用；但 refine 一次只报一个错且 code 固定 custom，多错误/分级用 superRefine + `fatal:true` + `return z.NEVER`。（文章109）
- **transform 里别 throw 做校验**：是 bad smell；transform 后想校验加工结果要 `.pipe(nextSchema)`。（文章110）
- **含 async 校验/变换必须用 parseAsync/safeParseAsync**，同步版本遇异步直接报错。（文章104、110）
- **.extend 是覆盖不是合并**：重名字段会被完全替换，原有 `.min(1)` 等规则被丢掉。（文章112）
- **.partial() 不递归**：嵌套对象字段仍必填，要递归用 `.deepPartial()`（注意 v4 已移除该方法，需手工逐层 partial）。（文章112）
- **input/output 混用**：只要 Schema 有 `default/transform/coerce/catch/preprocess/pipe` 就分叉，前后端用错类型会逼出满屏 `?.`。（文章111）
- **只导类型不导 Schema**：别的模块无法 parse，Hono RPC 类型链断裂。（文章111、113）
- **LLM parse 失败日志必带 raw**：没有原始字符串，事后排查模型漂移基本没戏。（文章114）
- **版本差异**：专栏基于 Zod v3（`^3.24.1`）；v4 中 `z.string().email()` 改为顶层 `z.email()`、错误格式化 API 有变、`deepPartial` 移除。（文章102、112）

### ## 与本项目(AI-Companion)的关联 + 下一步动手建议

本子主题是单元5「工程底座」里最直接服务 AI Companion 的一环。AI 陪伴 Agent 的不可信数据边界极多：用户 prompt、对话上下文、LLM 结构化输出、工具/MCP 参数、环境变量（API Key/模型名/温度上限）、RAG 检索元数据——Zod 正是把这些边界契约收敛成一份 Schema 的工具，避免在业务层散落 `if (typeof x...)`、`as any`。后续单元8 的 Vercel AI SDK（116+）也大量依赖 Zod 定义 tool/structured output schema，这里是前置基础。

下一步动手建议：

1. **建共享 schema 包**：在项目里建 `packages/shared/src/schemas/`（或 `src/schemas/`），把 `chat.ts`（MessageSchema/ChatRequestSchema/AIReplySchema/ChatResponseSchema）、`common.ts`（ApiErrorSchema/ApiOkSchema 工厂/TimestampsSchema）、`env.ts` 落地，作为全站契约层。
2. **抄两个工具函数**：把上面的 `validate`（Hono 统一校验封装）和 `LLMJsonSchema`（脏输出防御层）直接搬进 `server/lib/`，后续所有路由和 LLM 调用复用。
3. **给 Chat 接口接全链路**：入口 `validate('json', ChatRequestSchema)` → 业务层信任入口不重复校验 → LLM 输出 `LLMJsonSchema(AIReplySchema).safeParse` + 失败打日志带 raw → 出口 `ChatResponseSchema.parse`；前端用 `z.input` 类型 + Hono RPC client，响应再 parse 一次。
4. **环境变量上 Zod**：写一个 `EnvSchema`（`z.coerce.*` + `.default()`），启动时 `EnvSchema.parse(process.env)` 一次性校验 API Key/模型名/温度上限，让运行时配置错误在启动期暴露而非线上炸。
5. **工具/MCP 参数统一**：每个工具的参数定义为一份 Zod Schema，`zodToJsonSchema` 喂模型 + `parse` 校验模型返回 + `z.infer` 给 TS，一份管三件事。
6. **验证习惯**：按全局规范，改了 ChatRequestSchema 字段/校验后默认跑测试或真实页面联调，确认前端表单提示 + 后端 400 + 类型同步生效（用「maxTokens 上限改一行全链路跟上」作为冒烟用例）。
