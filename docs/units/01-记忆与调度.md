# 单元1 · 记忆与调度

> 精读文章：04 内存调度思考、06 向量化与语义检索、07 情绪状态机设计、09 混合记忆架构设计

> 项目锚点：本项目阶段1已落地记忆系统三张表（`agent_conversations` / `agent_conversation_messages` / `agent_memories`）与 Prompt 组装（`build-prompt.ts`），本文用它们作为"已建好的地基"来对照课程方案。

## 核心概念：每条一句话讲清

- **无状态矛盾**：LLM 每次推理都是无状态的"有限窗口概率机器"，但 AI 伴侣是"长期关系型 Agent"，记忆/阶段/情绪连续性必须由应用层调度补上（文章04）。
- **内存调度器（认知路由器）**：决定每轮"什么进 prompt、什么压缩、什么长期存储、什么丢弃"，并按场景（闲聊/争吵/安抚/亲密升级）切换不同记忆策略（文章04）。
- **OS 内存类比**：LLM=CPU（上下文窗口=寄存器/L1L2）、短期记忆=RAM、向量库/关系库=硬盘；调度器=OS 内核，负责硬盘↔RAM 之间的"页置换"（文章04）。
- **任务三优先级**：P0 实时交互链路（必须毫秒级秒回）、P1 关键状态更新（下一轮对话前必须生效）、P2 记忆整理（后台异步，可夜间批处理）（文章04）。
- **必要的阻塞**：一致性阻塞（改名字/忘记某事时，先更新记忆再回复）、上下文溢出阻塞（Token 触顶先压缩/遗忘再处理新消息）（文章04）。
- **Embedding（嵌入）**：把文本映射成高维向量，"意思相近"→"数字相近"；常见维度 768（bge-base）/1536（text-embedding-3-small）（文章06）。
- **余弦相似度**：AI 伴侣场景的首选度量，只看向量方向不看长度，因此长短句只要语义相同得分就高；欧氏距离受长度影响不适用（文章06）。
- **语义检索闭环**：记忆产生 → 切分提炼 → 向量化 → 存向量库（带元数据）→ 查询向量化检索 Top-K → 注入上下文（文章06）。
- **纯向量检索的三个盲区**：精确事实（生日几号）、时间范围（上周聊了啥）、专有名词（《三体》），都需要"精确/结构化"而非"意思相近"（文章09）。
- **混合检索架构**：语义检索 + 结构化查询 + 时间范围查询 + 关键词检索四通道并行，覆盖所有查询类型（文章09）。
- **记忆按访问模式分层存储**：KV 存高频热数据（短期上下文/情绪/亲密度）、D1 存需条件查询的结构化数据（画像/摘要/记忆条目）、Vectorize 存需语义检索的文本（文章09）。
- **双写点**：事件记忆同时写 Vectorize（语义）和 D1（关键词），是混合架构唯一的存储冗余代价（文章09）。
- **情绪状态机（FSM）**：在应用层维护一个独立于 LLM 的情绪状态，由用户行为驱动转移，每轮把当前情绪注入 Prompt——LLM 只负责"演"，不负责"决定演什么"（文章07）。
- **情绪三件套边界**：时间衰减（半衰期 + 最短持续时间）、冲突解决（优先级 + 缓解优先于制造新冲突）、防抖动（冷却期）（文章07）。
- **亲密度联动**：亲密度（0-100）既被情绪事件反向更新，又会改变情绪转移阈值（越亲近越容易吃醋也越容易被哄好）（文章07）。

## 设计决策与"为什么"

- **为什么不全量喂历史给 LLM**：Token 有上限、Token 越多越贵越慢、模型不区分"重要 vs 闲聊"会随机记忆——所以必须加调度层（文章04）。
- **为什么放弃滑动窗口（最近 N 条）**：会丢早期关键信息（第1天说的名字，第100天问就忘了）。调度器要能跨越窗口精准召回长期记忆（文章04）。
- **为什么用余弦而非欧氏**：用户表达有长有短，关心"意思像不像"而非"字数齐不齐"，余弦只看方向天然契合（文章06）。
- **为什么切分提炼而非整段存**：短文本语义更聚焦→检索更准，且更省存储；粒度推荐"一个完整事件/一个明确事实"为一条（文章06）。
- **为什么选 Vectorize 而非 Pinecone/Milvus**：边缘部署离用户近、与 Workers AI 同生态原生集成、冷启动低、起步免费；AI 伴侣是中小规模（百万级向量）正好够用，检索需控制在 50ms 内边缘优势关键（文章06）。
- **为什么要查询理解（多一次 LLM 调用）**：用 100-300ms 的延迟成本换"只激活需要的通道"，避免每次都全通道扫库；用户量大时省下的后端资源远超这点延迟（文章09）。
- **为什么用 FSM 而不让 LLM 自判情绪**：FSM 可预测（确定性转移，不会随机抑郁）、可调试（状态/条件一目了然）、可持久化（一个字符串+几个数值）、低成本（纯逻辑零推理）（文章07）。
- **为什么"缓解优先于制造新冲突"**：用户说"对不起，不该和小李吃饭"既是道歉又提了异性，应优先消解生气而非顺势吃醋——解决冲突比制造新冲突重要（文章07）。
- **为什么不同情绪衰减时间不同**：生气半衰期 3h、开心 1h、害羞 30min——更符合人类心理；再加 `minDuration` 防止"刚生气就秒回平静"的不自然（文章07）。
- **为什么结构化查询权重最高（1.2x）**：从画像表查出的生日是 100% 准确事实，不存在"相似度"，融合时应压过语义检索（1.0x）/时间（0.9x）/关键词（0.85x）（文章09）。
- **为什么事实"覆盖写"、事件"追加写"**：画像表要永远是最新状态（换工作覆盖 job 字段），事件表要保留完整历史轨迹（离职+入职各存一条）（文章09）。
- **为什么写入分同步/异步**：KV 短期上下文必须同步（下一轮立即要读），D1/Vectorize 的 LLM 抽取走队列异步（不阻塞用户秒回）（文章04、09）。

## 关键代码/片段

**1. 情绪转移规则表 + 衰减（FSM 骨架，文章07）**

```typescript
// 转移规则：calm 受事件触发进入各情绪；angry 有多种退出路径
const transitionRules = {
  calm: { triggers: [
    { event: 'praise',   nextState: 'happy' },
    { event: 'argument', nextState: 'angry' },
    { event: 'rival',    nextState: 'jealous' },
  ]},
  angry: { exitConditions: [
    { event: 'apology', nextState: 'calm' },
    { event: 'gift',    nextState: 'happy' },  // 可跨非平静态直接跳
    { event: 'decay',   nextState: 'calm' },   // 超时衰减
  ]},
}

const DECAY_CONFIG = {
  angry: { halfLife: 3 * 3600_000, minDuration: 30 * 60_000 },
  happy: { halfLife: 1 * 3600_000, minDuration: 10 * 60_000 },
}
// 半衰期指数衰减：minDuration 内不衰减，强度 < 10 归零回平静
function calculateDecay(mood: string, ts: string, intensity: number) {
  const elapsed = Date.now() - new Date(ts).getTime()
  const c = DECAY_CONFIG[mood]
  if (elapsed < c.minDuration) return intensity
  const next = Math.round(intensity * Math.pow(0.5, elapsed / c.halfLife))
  return next < 10 ? 0 : next
}
```

**2. 情绪冲突解决：缓解优先（文章07）**

```typescript
const EMOTION_PRIORITY = { angry: 5, sad: 4, jealous: 3, shy: 2, happy: 1, calm: 0 }
function resolveConflict(currentMood: string, candidates: {state: string, intensity: number}[]) {
  // 当前是高优先级负面情绪时，优先选"降级"候选（如道歉），而非制造新情绪
  const deescalation = candidates.find(c => EMOTION_PRIORITY[c.state] < EMOTION_PRIORITY[currentMood])
  if (deescalation) return deescalation
  return candidates.sort((a, b) => EMOTION_PRIORITY[b.state] - EMOTION_PRIORITY[a.state])[0]
}
```

**4. 结果融合：来源加权 + 时效加成（文章09，伪代码提炼）**

```typescript
const SOURCE_WEIGHT = { structured: 1.2, semantic: 1.0, temporal: 0.9, keyword: 0.85 }
function finalScore(raw: number, source: keyof typeof SOURCE_WEIGHT, ageMs: number) {
  let s = raw * SOURCE_WEIGHT[source]
  if (ageMs < 24 * 3600_000) s *= 1.10      // 24h 内 +10%
  else if (ageMs < 7 * 24 * 3600_000) s *= 1.05 // 7d 内 +5%
  return s
}
// 去重用字符级 Jaccard/编辑距离（>85% 视为同一条，留高分），无需再走向量
```

## 踩坑点：易错点与边界

- **并发一致性**：用户连发"我出门了""忘带钥匙"两条，若并行处理第二条会丢第一条上下文，甚至分别回复造成错乱——必须用会话锁/消息队列保证同一用户串行（文章04）。本项目 `agent_conversations` 对 `(user_id, agent_id)` 建了唯一索引，是会话级串行的天然抓手。
- **时间感知**：LLM 不知道"现在"。用户说"上周的事"要在写入时换算成绝对时间戳（如 2024-10-15）存储，否则未来唤醒时"上周"失去意义（文章04）。
- **遗忘权（合规）**：用户要"删掉所有关于我的"必须物理硬删，能按 userId/sessionId 精确清理向量库 + 关系库，不能只逻辑屏蔽（文章04）。本项目 `agent_memories.status` 有 `deleted` 态，注意它是逻辑删除，合规场景要补真正的 hard delete。
- **专有名词翻车**：Embedding 易把"三体"映射到"物理/科幻"方向而非精确命中两个字——专有名词靠关键词检索（D1 LIKE/FTS）兜，不能只靠向量（文章09）。
- **时间范围别查原始对话**：直接捞"上周 200 条原始消息"会撑爆上下文，时间通道应检索后台异步生成的"对话摘要"，而非原文（文章09）。
- **截断是必需的**：融合后取 Top-5~8 即可，塞太多记忆会稀释重要信息反而降低回复质量（文章09）。
- **去重别用向量**：检测"几乎相同文本"用字符级 Jaccard/编辑距离就够，目的不是"语义相近"而是"文本重复"，走向量是浪费（文章09）。
- **情绪秒切问题**：连发消息时情绪不应 1 秒内 开心→生气→害羞 反复横跳，需冷却期（Cooldown，如 5s）+ `minDuration` 双重约束（文章07）。
- **双写的隐性代价**：事件记忆 D1/Vectorize 双写，要保证两边不漂移（写失败一边、删只删一边都会导致召回不一致），需要在写入/删除路径上做好原子性或补偿。

### 下一步动手建议（按投入产出排序）

1. **补全记忆抽取的 LLM 实现**（最高优先，TODO 已标）：`memory-extraction.ts` 把关键词兜底替换为 LangChain 结构化输出，按文章09 拆成两个独立调用——**事实提取（覆盖写，键值对）vs 事件提取（追加写，自然语言）**。当前 `agent_memories` 单表混存，建议给 `type` 明确区分 fact/event/preference，或考虑事实另立画像表（覆盖写语义更干净）。
2. **激活会话摘要（时间范围通道雏形）**：`agent_conversations.summary` 字段已留位，接一个"每 N 轮异步压缩"的后台任务填它，并在 `build-prompt.ts` 中已经会注入（`summary` 分支已写好），属于低成本高回报。
3. **引入情绪状态机（文章07）**：给 `agent_conversations` 或新表加 `npc_mood / mood_intensity / mood_timestamp / intimacy_level` 字段（情绪天然是热数据，课程建议放 KV，本项目若暂无 KV 可先落 D1）。在主链路前插一个"情绪分类器 → 转移规则 → 注入 Prompt"步骤，与现有 `build-prompt.ts` 的 system 段拼接即可，无需大改。
4. **补向量层（文章06/09）**：当 `agent_memories` 条数变多、关键词召回不够时，再引入 Vectorize 做语义检索，对事件类记忆做 **D1+Vectorize 双写**。注意先解决遗忘权的级联硬删（向量库 + D1 一起删）。
5. **加会话锁/串行保证**（文章04 并发一致性）：利用已有的 `(user_id, agent_id)` 唯一会话，在写消息路径上保证同一会话消息串行处理，避免连发错乱。

### 一句话呼应

本项目阶段1把课程"调度层 + 三层记忆"中**最实时、最影响体验的两块（短期消息流水 + 长期记忆条目 + Prompt 分拣）**先落地了；情绪状态机、会话摘要、向量语义层是按"体验收益"梯度往后补的清晰路线，且现有 schema 与 `build-prompt.ts` 已为它们预留了字段和注入位（summary 分支、memories 标注），扩展成本低。
