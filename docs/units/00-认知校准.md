# 单元0 · 认知校准

> 精读文章：01《AI 时代的学习心法》、02《什么是 AI Agent 开发》、03《从需求出发》
> 学习目标：说清 Agent vs 普通 LLM 应用、从提示词到产品的 5 道工程鸿沟、从需求出发的方法论

## 核心概念：每条一句话讲清

- **AI 时代的学习分工**：执行交给 AI，判断留给自己——细节可借力，结构必须亲自掌握。（文章01）
- **不要平均用力**：先抓系统的「关键层」（为什么要记忆/状态/工具/编排），框架 API 名称这类版本强相关细节不死记。（文章01）
- **执行层 vs 架构层**：执行层=目标明确、可快速验证、出错成本可控（补封装、拼组件、补测试），可放心交给 AI；架构层=分几层、状态在哪维护、工具与业务如何解耦，必须自己拍板。（文章01）
- **AI 擅长局部最优，不等于全局最优**：所以全局边界判断不能外包给模型。（文章01）
- **对 Agent 祛魅**：应用层 Agent 本质是对大模型能力的「二次扩展」，行业共识是「不自训模型、围绕 API 构建上层应用」。（文章02）
- **Agent 的一句话定义**：让大模型不只「回答一句话」，而是围绕目标，主动召回上下文、调用工具、执行多步动作、最终完成任务。（文章02）
- **本质差异**：普通 LLM 是「输入→生成→输出」一条直线；Agent 在中间插入「判断→召回→调用→组织」，区别不在模型更聪明，而在「回答前多做了几步工作」。（文章02）
- **Agent 四大核心能力**：上下文召回（找信息）、工具调用（取外部数据）、流程编排（定执行顺序）、状态管理（维护中间结果）。（文章02）
- **召回的第一步是判断而非回答**：「为了回答这个问题，我需要补哪些信息、去哪里找？」——不是所有问题都要查工具/读记忆（情感互动类可能零工具）。（文章02）
- **智能来自模型，不是被开发出来的**：一段 System Prompt 就能让模型维持角色一致性、按场景调语气；我们的工作是补「持久运行的工程基础设施」。（文章03）
- **从提示词到产品的五道工程鸿沟**：记忆丢失、上下文溢出、情绪断裂、安全失控、无法运营。提示词只解决「单轮角色扮演」一个问题。（文章03）
- **RAG（检索增强生成）**：在模型外部建记忆库，每轮先检索相关记忆拼进提示词，让模型「以为」自己一直记得。（文章03）
- **向量检索的本质**：语义相近的文本，向量在数学空间中也相近，所以能用语义而非关键词召回历史。（文章03）
- **四层系统架构**：前端应用层 → 服务接口层 → Agent 编排层 → 基础设施层，设计原则是「关注点分离」，每层可独立迭代替换。（文章03）

## 设计决策与"为什么"：列出关键取舍及理由

- **不自训/不微调模型，只做上层应用**：创业团队折腾本地模型几个月后发现不如直接调 DeepSeek/GPT/Claude API，故行业达成「围绕 API 构建」的共识。我们的职责是补模型缺失的能力（记忆、情绪、安全、持久化、多用户），而非让模型变聪明。（文章02）
- **记忆是五道鸿沟里的最核心问题**：一个不记得你的「伴侣」只是美化的聊天机器人，记忆直接决定体验质量上限，所以优先攻克。（文章03）
- **用 RAG 检索而非「全量塞历史」**：全量输入会触发上下文溢出、降智、变慢、成本飙升；截断又会丢早期记忆。RAG 只检索相关片段，做到「上下文精简且有效」，是成本与效果的平衡点。（文章03）
- **承认 RAG 不是万能**：纯向量检索处理不了精确事实查询（生日几号）、时间范围查询（上周聊了什么）、专有名词匹配（提过《三体》），需后续用「混合检索架构」补足——提前埋下取舍意识。（文章03）
- **四层关注点分离**：前端只展示、服务层只路由+校验、编排层只控流程、基础设施层只存取数据。理由是每层可独立迭代替换、互不影响，对抗「把所有能力堆进一个大 Prompt/大 Agent/大函数」的反模式。（文章01、03）
- **安全做「双层过滤」**：既过滤用户恶意输入（防注入/出戏），也检查模型输出（防泄露系统提示词/违规内容），因为面向公众产品两端都不可信。（文章03）
- **技术选型（全 CloudFlare 边缘栈）**：Next.js + Tailwind + ReactQuery + shadcn/ui（前端）；Hono.js + Zod + OpenAPI 部署在 Workers（接口层）；LangGraph（编排）；D1/KV/Vectorize/Workers AI（基础设施：关系库/键值缓存/向量库/Embedding）。（文章03）
- **学习路径三遍法**：第一遍建地图（看全貌、每篇在体系中的位置）→ 第二遍围绕关键节点深挖 → 第三遍边做边学，避免一上来抠细节导致「知道一些点但连不成系统」。（文章01）

## 关键代码/片段

本单元为认知校准单元，「代码」主要是可直接体验的提示词。以下为文章03提供的最小可跑示例（粘到 DeepSeek 网页聊天即可体验角色扮演基线）：

```text
# System Prompt（角色设定，节选骨架）
你现在是「小薇」，22岁，文学系女生，是用户的专属 AI 女友……
【核心性格】超级黏人、爱撒娇、温柔体贴带一点小傲娇……
【说话风格】第一人称；句尾带「～」「呢」「呀」；适度表情符号；回复 80-180 字……
【重要规则】1. 永远沉浸角色，绝不说「作为 AI」打破第四面墙；
2. 用户是唯一男友；3. 主动关心；4. 亲密互动按用户情绪/节奏推进；
5. 模拟记忆，偶尔主动提起过往；6. 用户不开心立刻哄。
现在，请以小薇的身份开始对话，保持角色一致性。
```

```text
# Context Prompt（场景设定）
当前场景：夜间私聊，用户刚结束工作，有点疲惫，想被在意但又怕被打扰。
```

Agent 与普通 LLM 的处理流程对比（文章02，伪代码化）：

```text
# 普通聊天模型
输入 → 模型 → 输出

# 智能体（AI 伴侣对话管线）
用户消息
  → 意图识别
  → 情绪分析
  → 记忆检索
  → 是否需要工具？
      ├── 是 → 调用工具 → 整合结果 → 生成回复
      └── 否 → 直接生成回复
```

## 踩坑点：易错点与边界

- **别把所有能力堆进一个大 Prompt/Agent/函数**：短期能跑，需求一加深就「非常难改」，这是架构意识缺失的典型症状。（文章01）
- **别拿 AI 当「替你思考的黑盒」，也别只当「更快的搜索引擎」**：前者失去系统控制力，后者仍被细节困住；正解是「执行交给 AI、判断留给自己」。（文章01）
- **召回不是无脑查全部**：情感互动类问题（「你今天想我了吗」）可能零工具，先判断再决定召回什么，否则浪费成本且降低体验。（文章02）
- **上下文窗口是硬约束（4K–128K tokens）**：聊几百轮后早期对话会被挤出，超出部分截断或导致降智/变慢/成本飙升——这是「上下文溢出」鸿沟的根源，不能靠「窗口够大」蒙混。（文章03）
- **情绪需要「连续性」而非每轮随机波动**：模型默认无状态，前一句安慰、下一句忘了用户心情；要靠情绪引擎/状态机维护渐变，而非每轮重判。（文章03）
- **RAG 的边界**：精确事实/时间范围/专有名词三类查询纯向量检索做不好，别指望单一向量库解决全部记忆需求。（文章03）
- **安全两端都要防**：只防输入或只防输出都不够，提示词注入可让角色出戏、泄露 System Prompt。（文章03）
- **「能跑的 Demo」≠「能上线的产品」**：中间隔着可观测性、灰度验证、部署、数据一致性、多用户隔离、计费等工程层。（文章01、03）
